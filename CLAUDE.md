## Status Project

Ini adalah **NEXA**, aplikasi internal untuk XNet (PT. XPlus Network Indonesia) yang akan menjadi tulang punggung administrasi/operasional untuk perusahaan ISP: manajemen pelanggan, katalog produk/layanan, billing, dan pembayaran via Xendit, yang selanjutnya akan berkembang ke integrasi jaringan (MikroTik, OLT) dan operasional ISP secara lebih luas.

Repository ini masih tahap awal. Modul domain yang sudah nyata (bukan sekadar shell) adalah **Authentication** (login phone + OTP WhatsApp), **User** (CRUD admin untuk seluruh akun — customer, staff, teknisi, admin, bukan cuma customer), **Product & Package** (katalog produk + paket bundling), **PoP & Coverage** (master data infrastruktur jaringan — titik PoP + area cakupan layanan), **Service** (instance langganan pelanggan — alamat pemasangan + koneksi ke coverage), dan **Sales** (transaksi/order produk terhadap sebuah service) — lihat "Authentication / Login", "User", "Product & Package", "PoP & Coverage", "Service", dan "Sales" di bawah untuk detail implementasi. Modul bisnis lain (Installation, Billing, Xendit, dst.) belum dibangun. `README.md` adalah dokumen perencanaan arsitektur yang hidup (living document, dalam Bahasa Indonesia) yang menjelaskan arah project; jadikan itu sebagai sumber kebenaran untuk struktur dan konvensi yang dituju, sampai kode yang benar-benar berjalan menetapkan sebaliknya.

## Arah Arsitektur (dari README.md)

Project ini bermaksud mengikuti konvensi berikut saat kode domain mulai ditambahkan — terapkan ini saat membuat fitur baru, jangan default ke controller gemuk:

- **Layering**: Form Request untuk validasi, Service Layer untuk business logic, Eloquent untuk persistence. Jauhkan business logic dari Controller; gunakan dependency injection.
- **Konvensi penamaan** per konsep domain, contoh untuk entitas `Customer`: `CustomerController`, `CustomerService`, `CustomerPolicy`, `CustomerRequest`.
- **API**: aplikasi customer-facing (web/Android/iOS) akan mengakses lewat REST API versioned di `/api/v1`, terpisah dari admin UI berbasis Blade.
- **Pembagian teknologi frontend** — gunakan teknologi sesuai tanggung jawabnya, jangan default pakai Vue:
    - Blade: layout, dashboard, halaman CRUD, form, table.
    - Alpine.js: interaksi ringan (modal, dropdown, accordion, toast, toggle).
    - Vue: hanya untuk komponen yang benar-benar kompleks (dashboard realtime, monitoring jaringan, grafik, wizard multi-step).
- **Integrasi jaringan**: integrasi MikroTik dan OLT (HSGQ) direncanakan di belakang pola adapter/driver supaya vendor baru bisa ditambahkan tanpa menyentuh modul bisnis.
- Soft delete hanya jika benar-benar dibutuhkan, bukan default; timestamp standar Laravel; foreign key dan index yang sesuai diharapkan ada di semua skema.

### Authentication / Login

Sudah **diimplementasikan sungguhan** (bukan lagi shell/stub) — login pakai nomor telepon + OTP WhatsApp, lengkap dengan migration, service layer, dan middleware auth.

- Login menggunakan **nomor telepon**, bukan email/password — berbasis OTP, bukan login password tradisional untuk end user.
- OTP dikirim melalui **WhatsApp**, bukan SMS.
- WhatsApp gateway adalah **service terpisah** (`go-whatsapp-web-multidevice`), diakses lewat REST API. NEXA tidak berkomunikasi langsung dengan WhatsApp/Meta.
- NEXA sendiri yang memegang lifecycle OTP — pembuatan, penyimpanan, masa berlaku, dan validasi semuanya terjadi di aplikasi ini, bukan di service WA gateway.
- **Tidak ada auto-register.** Nomor yang belum ada di tabel `users` ditolak saat minta OTP (`SendOtpRequest`, rule `exists:users,phone`) — pembuatan akun customer adalah tanggung jawab modul Customer yang belum dibangun. Untuk dev lokal, `database/seeders/AdminUserSeeder.php` menyediakan satu akun admin (`phone` `6281234567890`) via `php artisan migrate:fresh --seed`.
- **Format nomor telepon: selalu disimpan dalam format internasional (kode negara `62` + nomor, tanpa `0` di depan), kolom `phone` bertipe `unsignedBigInteger` (BIGINT UNSIGNED) — bukan string/VARCHAR.** Keputusan sadar: nomor Indonesia dianggap selalu diawali kode negara 62 begitu dinormalisasi, jadi tidak ada leading-zero bermakna yang perlu dijaga (leading `0` yang ada di input user hanyalah trunk prefix domestik yang dibuang, bukan bagian dari nomor kanonis). Field login di `resources/views/auth/login.blade.php` tetap **satu input polos** (placeholder `81234567890`, tanpa badge/prefix visual `+62`) — user cukup ketik nomor lokal tanpa `0`/`62`, backend yang menormalisasi. Normalisasi dilakukan oleh `App\Support\PhoneNumber::normalize()` (dipanggil dari `SendOtpRequest::prepareForValidation()`): input dibersihkan dari karakter non-digit, lalu kalau diawali `0` diganti jadi `62`, kalau sudah diawali `62` dibiarkan, selain itu `62` ditambahkan di depan. Pakai helper ini lagi (jangan re-implement) di modul lain yang menerima input nomor telepon (mis. nanti Customer). `User::phone` di-cast `'integer'` di model.
- **Skema**: migration `..._add_phone_admin_last_login_at_to_users_table` menambah `phone` (unsignedBigInteger, nullable, unique — nullable supaya tidak konflik dengan baris user yang belum punya phone; "harus terdaftar" ditegakkan di Form Request, bukan di DB), `admin` (boolean), `last_login_at` ke `users`. Kolom domain customer dari draft drawio (`ktp`/`dob`/`gender`/`code`) **sengaja ditunda** ke migration terpisah saat modul Customer dikerjakan. Tabel baru `otp_codes` (`app/Models/OtpCode.php`) — `user_id` (FK, bukan kolom `phone` terpisah, karena OTP hanya bisa diminta untuk user yang sudah ada), `code_hash` (HMAC-SHA256, bukan bcrypt — OTP berumur pendek/sekali pakai dan sudah dilindungi rate limit + attempt lockout), `expires_at`, `consumed_at`, `attempts`.
- **Service layer**: `app/Services/OtpService.php` (generate/simpan/expire/validasi kode, cooldown resend 60 detik via `Cache`, lockout setelah `OTP_MAX_ATTEMPTS` percobaan salah), dipanggil dari `LoginController` (sudah bukan stub lagi, di-inject `OtpService`).
- **WhatsApp gateway pakai pola driver/adapter** (`app/Services/Whatsapp/`): interface `WhatsappGateway`, driver default **`log`** (`LogWhatsappGateway` — tulis OTP ke `storage/logs/laravel.log`, dipakai karena server `go-whatsapp-web-multidevice` sungguhan belum tersedia di dev), dan skeleton `HttpWhatsappGateway` (raw `Http::` facade sesuai mandat README, endpoint masih placeholder — belum dikonfirmasi ke server nyata). Pilihan driver dari `config('services.whatsapp.driver')` / env `WHATSAPP_GATEWAY_DRIVER`, di-bind di `app/Providers/WhatsappServiceProvider.php` (terdaftar di `bootstrap/providers.php`).
- **Pengiriman OTP lewat job `app/Jobs/SendOtpWhatsappNotification.php`, tapi di-dispatch di koneksi queue `sync`** (`config('otp.queue_connection')` / env `OTP_QUEUE_CONNECTION`, default `sync`), **bukan ikut `QUEUE_CONNECTION=database` global**. Ini keputusan sadar, bukan default Laravel: percobaan pertama pakai `database` sempat menyebabkan job nyangkut di tabel `jobs` tanpa pernah diproses karena tidak ada `queue:work`/`queue:listen` yang jalan di dev — OTP jadi tidak pernah benar-benar "terkirim" (tidak ke-log), padahal baris OTP di DB sudah dibuat, bikin bingung ("kode di log kok tidak cocok" — padahal log-nya memang belum pernah ditulis, itu log basi dari percobaan sebelumnya). `sync` membuat job jalan inline saat request tanpa butuh worker terpisah. Kalau nanti `HttpWhatsappGateway` sungguhan dipakai di production dan panggilan HTTP-nya berpotensi lambat/nge-block request, pertimbangkan ubah `OTP_QUEUE_CONNECTION` kembali ke `database` (atau koneksi queue khusus) — jangan asumsikan `sync` cocok selamanya.
- **Routing & middleware**: `/login` + `/login/otp` dalam grup `guest`; `/dashboard` + `POST /logout` (baru) dalam grup `auth`. `bootstrap/app.php` pakai `redirectGuestsTo('/login')`/`redirectUsersTo('/dashboard')` (built-in Laravel 13). Rate limiter `otp-request` (3/menit per IP+phone) dan `otp-verify` (10/menit per IP) didaftarkan di `AppServiceProvider::boot()` — karena tidak ada `RouteServiceProvider` terpisah di skeleton Laravel 13 ini.
- Tombol "Keluar" di `header.blade.php` sudah di-wire ke `POST /logout` (form + `@csrf`), menggantikan placeholder `<a href="#">`.
- Test: `tests/Feature/Auth/LoginFlowTest.php` — pakai fake `WhatsappGateway` (bind instance yang menangkap kode asli) untuk membaca kode OTP tanpa bergantung pada skema hash internal.

### User (dulu "Customer" — scope sengaja dilebarkan)

Modul kedua yang sudah nyata — CRUD admin lewat Blade di **`/users`** (list/create/edit/delete), belum ada endpoint `/api/v1` (menyusul saat customer app sungguhan mulai dibangun).

Awalnya dibangun sebagai modul "Customer" murni (`/customers`, model `Customer extends User` dengan global scope `admin=false`). **User lalu eksplisit meralat**: layar ini bukan cuma untuk kelola customer, tapi seluruh akun (customer, staff, teknisi, admin) — jadi model `Customer` **dihapus total**, diganti operasi langsung ke `App\Models\User`, dan route/URL pindah ke `/users`. Konvensi ke depan: **default ke istilah "user"/"pengguna", bukan "customer"/"pelanggan"**, kecuali fitur itu memang eksklusif milik customer (mis. nanti billing/invoicing yang nempel ke `sales`).

- **Satu tabel `users` untuk semua role**, dibedakan lewat role dari `spatie/laravel-permission` — lihat section "Authorization / Role & Permission" di bawah untuk detail. Tidak ada lagi kolom `users.admin` (sudah di-drop) maupun subclass `Customer` — Controller/Service/Request/Policy semua bertipe `App\Models\User` langsung.
- **Pilihan role** ada di form create/edit (`resources/views/users/_form.blade.php`, `<select name="role">`) — superadmin yang login bisa membuat/menandai role akun lain lewat UI yang sama, bukan cuma lewat `AdminUserSeeder`. **Guard yang sudah ada**: superadmin tidak bisa mengubah role akun miliknya sendiri keluar dari `superadmin` (`UserRequest::guardAgainstSelfDemotion()`) — supaya tidak keunci sendiri dari sistem selama belum ada mekanisme recovery lain. Guard granular lain (mis. mencegah superadmin biasa mengelola superadmin lain) **belum diimplementasikan** — user sudah bilang eksplisit itu "dipikirkan lagi nanti" sesuai kebutuhan; jangan improvisasi skema permission granular sebelum diminta.
- **`users.code`** hanya digenerate untuk role **`customer`** (`'CUS'.str_pad($user->id, 6, '0', STR_PAD_LEFT)` di `UserService::create()`, setelah insert — aman dari race condition karena mengandalkan auto-increment id) — role lain (`superadmin`/`technician`/`finance`/`sales`) dibiarkan `code = null` karena `code` itu untuk keperluan invoicing/billing customer (lihat draft ERD), bukan staff.
- **`user_details`** — tabel ekstensi 1:1 dengan `users`, **shared primary key** (`user_details.id` = `users.id`, FK cascade delete, bukan auto-increment sendiri), **berlaku untuk semua role** (bukan customer-eksklusif — makanya relasi `userDetails()` didefinisikan di `User`). Kolom: `nik` (string 16 char, bukan `unsignedBigInteger` seperti `phone` — alasan normalisasi leading-zero yang melatarbelakangi keputusan `phone` tidak berlaku untuk NIK, diperlakukan sebagai identifier opak), `birth_date` (date), `gender` (string, nilai `'male'`/`'female'`), `ktp_photo` (string path — lihat "Foto KTP" di bawah untuk lokasi/akses). Model `App\Models\UserDetail` (`$incrementing = false`).
- **NIK bukan input bebas** — `gender` dan `birth_date` **selalu** diturunkan dari NIK lewat `UserDetail::parseNik(string $nik): ?array`, tidak pernah diinput manual. Struktur NIK: digit 7-8 = hari lahir (kalau > 40, kurangi 40 dan tandai `gender = 'female'`; kalau tidak, `'male'`), digit 9-10 = bulan, digit 11-12 = tahun 2 digit — abad ditentukan dengan mencoba `[abad sekarang, abad sekarang - 100]` dan pilih yang menghasilkan usia masuk akal (17–100 tahun relatif ke `now()`, dicek pakai `Carbon::diffInYears`). `checkdate()` PHP dipakai untuk validasi hari/bulan/tahun kabisat. Return `null` kalau tidak 16 digit, bulan/hari tidak valid, atau tidak ada kandidat abad yang menghasilkan usia 17–100 tahun — `null` berarti NIK ditolak di `UserRequest` (`UserDetail::parseNik($nik) === null` → error validasi "NIK tidak valid").
- **NIK dikunci setelah pertama kali tersimpan** — begitu `user_details.nik` sudah terisi, tidak bisa diubah lewat form manapun (termasuk form superadmin), ditegakkan di `UserRequest::guardNik()` (bandingkan NIK yang datang vs `$target->userDetails->nik` yang sudah ada; beda → error "NIK sudah terkunci"). Belum ada mekanisme override khusus untuk ini; kalau perlu koreksi, sementara lewat `tinker`/DB langsung. Form edit tidak lagi merender `<input>` untuk NIK begitu sudah terkunci — cuma teks statis + `<input type="hidden">` supaya tetap ikut ter-submit tapi tidak bisa diketik ulang (pertahanan UI; validasi server tetap penegak utama — **wajib** render `@error('nik')` di luar percabangan terkunci/tidak, supaya percobaan tamper tetap dapat feedback, bukan gagal senyap).
- **Kolom `email` dan `password` di `users` tetap NOT NULL** (peninggalan scaffolding default Laravel) walau login selalu lewat OTP dan tidak pernah memakainya. `UserService::create()`/`update()` mengisi placeholder inert: `password` → `Hash::make(Str::random(40))`, `email` → `"user-{$phone}@nexa.internal"` (diturunkan dari `phone` yang sudah unique, ikut disinkronkan ulang saat `update()`).
- **Authorization**: `App\Policies\UserPolicy` (viewAny/view/create/update/delete → `$user->isSuperadmin()`) — lihat section "Authorization / Role & Permission" untuk detail role. Laravel auto-discover Policy lewat konvensi nama (`User` model ↔ `UserPolicy`), dipanggil dari `UserController::__construct()` via `$this->authorizeResource(User::class, 'user')`. `App\Http\Controllers\Controller` (base class) di-extend dari `Illuminate\Routing\Controller` (`$this->middleware()`) + `use AuthorizesRequests` supaya `authorizeResource()` jalan.
- **Service layer**: `app/Services/UserService.php` — `create()`/`update()` jalan dalam `DB::transaction()`, `delete()` hard delete.
- **Request**: satu `App\Http\Requests\UserRequest` dipakai untuk create & update, reuse `PhoneNumber::normalize()` di `prepareForValidation()`.
- Sidebar "Pengguna" (`resources/views/components/sidebar.blade.php`) mengarah ke `users.index` (dulu "Pelanggan" → `customers.index`).
- Test: `tests/Feature/UserManagementTest.php` — CRUD superadmin, code auto-generate hanya utk role `customer`, listing gabungan lintas role, guard self-demotion role, validasi unique phone/nik, NIK lock, role selain superadmin dapat 403, dan akses foto KTP privat.

### Authorization / Role & Permission

Dibangun setelah modul User, langsung menyelesaikan dua hal yang sengaja ditunda di sana: lock NIK belum punya override, dan CRUD `/users` cuma biner admin/bukan-admin.

- **Package**: `spatie/laravel-permission` (bukan hand-roll) — dipasang via `composer require` + `php artisan vendor:publish --provider="Spatie\Permission\PermissionServiceProvider"`, menerbitkan `config/permission.php` dan migration `create_permission_tables` (tabel `roles`, `permissions`, `model_has_roles`, `model_has_permissions`, `role_has_permissions`). Tidak pakai fitur teams (`permission.teams` tetap `false`, default) karena belum dibutuhkan.
- **Satu role per user** (bukan many-to-many) — cukup untuk kebutuhan saat ini, dipilih user secara sadar; package tetap mendukung upgrade ke multi-role nanti (`syncRoles()` dipakai bukan `assignRole()` doang di `update()`, supaya nanti gampang diperluas).
- **5 role**: `superadmin`, `technician`, `finance`, `sales`, `customer` — daftar final dari user, **bukan** set customer/staff/teknisi/admin yang sempat disebut di context `user_details` sebelumnya (itu istilah umum, bukan daftar role). Di-seed lewat `Role::firstOrCreate(['name' => ..., 'guard_name' => 'web'])`, baik di migration data (lihat poin berikutnya) maupun `database/seeders/RoleSeeder.php` (idempotent, didaftarkan di `DatabaseSeeder` **sebelum** `AdminUserSeeder` supaya `migrate:fresh --seed` di masa depan tetap benar).
- **Migrasi data `users.admin` → role, lalu drop kolom** — dua migration terpisah (urutan penting): `2026_07_09_005247_assign_roles_from_admin_flag.php` (baca `admin` boolean yang masih ada lewat `DB::table('users')`, assign role `superadmin` kalau `true` else `customer`, lalu **baru setelah itu** kolom di-drop) dan `2026_07_09_005247_drop_admin_from_users_table.php`. Sengaja dipisah dari schema-change supaya data-migration dan drop-column masing-masing punya `down()` yang jelas.
- **`User` model**: pakai trait `Spatie\Permission\Traits\HasRoles`, method `isSuperadmin(): bool { return $this->hasRole('superadmin'); }` — satu tempat convenience, dipakai di `UserPolicy` & `KtpPhotoController` (jangan sebar `hasRole('superadmin')` literal di banyak file, biar gampang diperluas kalau nanti perlu OR beberapa role).
- **Gate masih konservatif — cuma `superadmin`.** Permission granular per role (siapa boleh apa persis di modul mana) **belum didesain** — user eksplisit bilang itu dipikirkan sesuai kebutuhan nanti. Supaya tidak berimprovisasi bikin skema permission yang mengada-ada, iterasi ini cuma **menggeser** gate `isAdmin()` lama jadi `isSuperadmin()` — perilaku setara persis dengan sebelumnya (dulu cuma `admin=true` yang boleh akses `/users` & lihat KTP orang lain, sekarang cuma role `superadmin`). `technician`/`finance`/`sales` **belum** dapat akses apa pun ke `/users` — itu menyusul begitu kebutuhan per-role jelas. Jangan tambah permission granular tanpa diminta eksplisit.
- Belum ada tabel `permissions` yang benar-benar dipakai (infrastrukturnya ada dari package, tapi belum ada permission bernama yang di-assign ke role manapun) — dan belum ada UI untuk kelola role/permission itu sendiri (create role baru, assign permission ke role). Keduanya gap yang didokumentasikan, bukan lupa.

#### Foto KTP (private storage)

- Disimpan di disk `local` (`config/filesystems.php` sudah root ke `storage/app/private` secara default di Laravel 13 — bukan `storage/app/public`), subfolder `ktp/` (`$file->store('ktp', 'local')` di `UserService`). **Tidak** ada di disk `public`, tidak ke-link oleh `php artisan storage:link`, jadi tidak bisa diakses langsung lewat URL `/storage/...` sama sekali.
- Serving lewat `GET /secure/ktp/{user}` → `App\Http\Controllers\KtpPhotoController@show`, route name `secure.ktp`, terdaftar di grup middleware `auth`. Otorisasi inline (bukan Policy — cukup sederhana untuk 1 aksi): boleh diakses **pemilik akun sendiri** (`$viewer->id === $user->id`) **atau role `superadmin`** (`$viewer->isSuperadmin()`). Selain itu → 403. Kalau `ktp_photo` null → 404. Response pakai `Storage::disk('local')->response($path)` (inline, bukan `download()`, supaya `<img>` tag bisa render langsung).
- Di view, **selalu** pakai `route('secure.ktp', $user)`, jangan `asset('storage/...')` — pola lama (waktu masih disk `public`) sudah diganti di `resources/views/users/_form.blade.php`.

### Product & Package

Modul ketiga yang sudah nyata — katalog produk (`/products`) dan paket bundling produk (`/packages`), CRUD admin lewat Blade, prasyarat untuk modul Service/Billing/Sales yang akan datang. Struktur (Model/Policy/Request/Service/Controller) direplikasi persis dari pola modul User.

- **Penyimpangan sadar dari draft ERD**: `database/database_design.drawio` punya kolom `products.product_type_id` yang merujuk tabel `product_types` — tabel itu **tidak digambar sama sekali** di diagram (FK menggantung). Diputuskan **bukan** dibuatkan tabel lookup terpisah untuk iterasi ini; diganti kolom `products.type` (string), divalidasi `Rule::in` dengan daftar tetap: `perangkat`, `jasa`, `biaya`, `lainnya` (`App\Http\Requests\ProductRequest::TYPES`). Draft ERD belum diperbarui untuk mencerminkan ini — kalau nanti kebutuhan kategori produk berkembang (banyak atribut per kategori, dsb.), baru dipertimbangkan naik jadi tabel FK sungguhan.
- **`packages.is_starter`** (boolean, default `false`) — nama kolom diselaraskan ke konvensi Laravel dari `packages.starter` di draft ERD (yang tidak ada keterangan di diagram, dikonfirmasi maknanya langsung ke user). `true` berarti paket bisa dipilih saat **pendaftaran baru** (mis. promo gratis 3 bulan + gratis pasang + modem premium bundel jadi satu paket starter); `false` berarti paket hanya bisa dipilih pelanggan yang **sudah aktif** berlangganan (mis. paket upgrade/add-on). Diisi `Auth::id()` di `ProductService`/`PackageService`, bukan retrofit ke modul User yang sudah ada. Kalau pola ini terbukti berguna, pertimbangkan menambahkannya ke `users` lewat migration terpisah nanti — jangan diam-diam diseragamkan tanpa keputusan eksplisit.
- **`code`** (`PRD`/`PKG` + 6 digit id, `str_pad`) — **selalu** digenerate untuk setiap produk/paket (beda dari `users.code` yang cuma untuk role `customer`), karena setiap baris katalog memang butuh kode bisnis. Sama seperti `users.code`, kolom `code` di migration harus **nullable** (bukan cuma unique) — digenerate lewat `update()` setelah insert awal supaya bisa memakai id auto-increment, jadi insert pertama tidak menyertakan `code` sama sekali.
- **`package_product`** (pivot `packages`↔`products`) — punya kolom tambahan `quantity` dan `price` (snapshot harga produk saat dibundel ke paket, independen dari `products.price` yang bisa berubah setelahnya), plus unique constraint `['package_id', 'product_id']` (satu produk cuma boleh muncul sekali per paket — kalau butuh lebih dari 1 unit, naikkan `quantity`, bukan baris baru). Dikelola lewat relasi `BelongsToMany` biasa (`Package::products()` / `Product::packages()`, `withPivot(['quantity', 'price'])->withTimestamps()`) dan `sync()` di `PackageService` — tidak perlu pivot model kustom untuk kebutuhan ini.

- **Service layer**: `app/Services/ProductService.php`, `app/Services/PackageService.php` — `create()`/`update()`/`delete()`, isi `code`/`created_by`/`updated_by`, dan (khusus Package) `sync()` pivot produk.

### Service

Modul kelima yang sudah nyata — instance langganan pelanggan (`/services`): alamat pemasangan + koneksi ke `coverage` + `package` yang didaftarkan, CRUD admin lewat Blade. Prasyarat untuk Installation/Billing/Sales yang akan datang. Draft ERD (`database/database_design.drawio`) punya beberapa kolom yang maknanya sempat ambigu dan dikonfirmasi langsung ke user sebelum migration ditulis — jangan improvisasi ulang tebakan lama (PIN verifikasi CS, `residential` sebagai boolean) yang sempat salah duga di sesi perencanaan sebelum ini:

- **`pin`** — **bukan** PIN verifikasi CS, tapi **password PPPoE** (6 digit) yang dipakai autentikasi router pelanggan ke MikroTik. Beda karakter dari `pops.token`: staff instalasi perlu lihat/copy nilainya berulang kali saat setting modem, jadi **disimpan plain string** (kolom migration `string(6)`, bukan `text` + cast `encrypted` seperti `pops.token`) — enkripsi di sini cuma akan merepotkan tanpa manfaat nyata. **Auto-generate 6 digit acak** (`ServiceService::generatePin()`, `random_int` + `str_pad`) saat `create()` (field `pin` sengaja **tidak ada** di form create — `ServiceRequest` menandainya `nullable` untuk method selain PUT/PATCH), tapi **bisa diedit manual lewat form edit** (`ServiceRequest` mewajibkan `required|digits:6` saat PUT/PATCH) untuk staff reset PIN kalau perlu — `ServiceService::update()` mempertahankan `pin` lama kalau field edit dikosongkan (`$data['pin'] ?? $service->pin`, pola sama seperti `PopService::update()` mempertahankan `token`). Tidak ada validasi unique (bukan identitas global, cuma password lokal per service). Redirect setelah `store()` sengaja menyertakan PIN yang baru dibuat di flash `session('status')` supaya staff langsung tahu nilainya tanpa harus buka halaman edit lagi.
- **`residential_name`** — **bukan boolean**, tapi **nama perumahan/apartment/komplek** lokasi pemasangan (draft ERD menamainya `residential`, kolom yang salah diduga sebagai flag boolean di sesi perencanaan awal). **Di-rename ke `residential_name`** (string, nullable) supaya tidak membingungkan dibaca sebagai flag — pola yang sama seperti `packages.starter` → `is_starter`.
- **`deleted_at` pakai Eloquent `SoftDeletes` trait sungguhan** — **beda** dari semua modul lain (User/Product/Package/PoP/Coverage yang semua hard delete). Alasan: riwayat langganan pelanggan jarang boleh hilang permanen, beda dengan master data katalog/infrastruktur yang aman dihapus keras. `ServiceController::destroy()` (lewat `ServiceService::delete()`) jadi soft-delete — row tetap ada di DB, otomatis tersembunyi dari query default (index listing) berkat global scope `SoftDeletes`. Kalau menambah modul baru lain yang juga representasi riwayat/transaksi pelanggan (bukan master data), pertimbangkan pola yang sama; kalau cuma master data/katalog, tetap default ke hard delete seperti modul-modul sebelumnya.
- **`activated_at`/`expired_at`/`dismantled_at`/`canceled_at`** — kolom nullable timestamp biasa, **belum ada tombol/aksi transisi state machine** di iterasi ini (belum ada UI activate/dismantle/cancel) — sama seperti `pops.last_online_at` sebelum modul monitoring ada. Skema disiapkan duluan, logic-nya menyusul begitu modul Installation/Billing dibangun. Jangan improvisasi state machine/validasi transisi sebelum diminta eksplisit.
- **`user_id`** dibatasi harus user dengan **role `customer`** (bukan staff/superadmin) — divalidasi di `ServiceRequest::withValidator()` (query `User::find($id)->hasRole('customer')`, bukan rule bawaan Laravel karena butuh cek relasi role, bukan sekadar exists). Picker di form pakai **AJAX typeahead** (sama alasan seperti `subdistrict_id` di PoP — jumlah customer bisa besar) via endpoint `GET /services/customers/search` (`ServiceController::searchCustomers()`, scoped `User::role('customer')` dari `spatie/laravel-permission`, limit 20 JSON, digate `$this->authorize('viewAny', Service::class)`). **Hanya bisa memilih user yang sudah terdaftar** — tidak ada opsi buat pelanggan baru dari form ini (typeahead cuma mencari, tidak pernah membuat; kalau tidak ketemu di hasil pencarian, `user_id` tetap kosong dan validasi `required` menolak submit). **Query kosong (`q=''`) sengaja mengembalikan daftar browse (20 customer pertama diurutkan nama), bukan array kosong** — beda dari pola picker `subdistrict_id` yang cuma respons ke query non-kosong — supaya user bisa klik kolom kosong untuk langsung melihat daftar pelanggan (bukan wajib mengetik dulu), meniru pola picker service di `~/Webapp/xnet/app` yang jadi rujukan. Frontend (`services/_form.blade.php`) baru memulai pencarian bertahap (debounce 300ms) begitu user mengetik **minimal 3 karakter**; `@focus` pada kolom yang masih kosong (`customerId` belum terisi) langsung fetch daftar browse tanpa menunggu ketikan.
- **`package_id`** (FK ke `packages`, **`restrictOnDelete()`**) — **wajib dipilih saat mendaftarkan service baru**, dan **wajib paket yang `is_starter = true`** (paket yang memang bisa dipilih saat pendaftaran baru — lihat "Product & Package" di atas). Ditegakkan dua lapis: (1) `ServiceController::create()`/`edit()` cuma mengirim `Package::where('is_starter', true)->get()` ke `<select>` di view, jadi paket non-starter tidak pernah muncul di dropdown; (2) `ServiceRequest::withValidator()` tetap cek ulang `$package->is_starter` di server (bukan cuma andalkan filter dropdown) — pertahanan berlapis kalau ada yang tamper request langsung. Ini **penyimpangan sadar dari draft ERD**: diagram menaruh asosiasi service↔package di tabel `sales` (order) yang belum dibangun, tapi user eksplisit minta paket dipilih langsung saat menambah `service` (bukan menunggu modul Sales) — dicatat di sini supaya penyimpangan ini jelas kalau nanti modul Sales dibangun dan ada pertanyaan "kenapa `services` sudah punya `package_id` sendiri". `<select>` biasa di form (pola sama seperti `coverage_id`), bukan typeahead (jumlah paket starter realistis kecil).
- **`coverage_id`** tetap `<select>` biasa di form (pola sama seperti `pop_id` di `coverages/_form.blade.php`) — jumlah coverage realistis kecil, tidak butuh typeahead.
- **`code`** (`SRV` + 6 digit id) dan **audit trail `created_by`/`updated_by`** — pola yang sama seperti modul-modul sebelumnya (PoP & Coverage melanjutkan pola dari Product & Package, ini melanjutkan lagi).
- **Label "Wilayah" di picker subdistrict sempat salah ditulis "(Kecamatan)"** — dikoreksi jadi **"(Kelurahan)"** di `pops/_form.blade.php` dan `services/_form.blade.php`. `subdistricts.name` (kolom yang benar-benar dicari `LIKE` oleh endpoint search) adalah level **kelurahan/desa**, bukan kecamatan — kecamatan ada di kolom `subdistricts.district_name` satu level di atasnya (lihat urutan tampil hasil pencarian: `name, district_name, city_name, province_name` = kelurahan, kecamatan, kota/kabupaten, provinsi, dari granular ke luas). Kalau menambah label baru yang merujuk pencarian ini, pakai "Kelurahan", bukan "Kecamatan".
- **Refactor endpoint search subdistrict jadi controller mandiri** — sebelum modul ini, `GET /pops/subdistricts/search` cuma dipakai `pops/_form.blade.php` lewat `PopController::searchSubdistricts()`. Sekarang `services/_form.blade.php` butuh picker subdistrict juga, jadi logic-nya diekstrak ke `App\Http\Controllers\SubdistrictController@search`, route pindah ke top-level `GET /subdistricts/search` (nama `subdistricts.search`, menggantikan `pops.subdistricts.search`) supaya tidak terasa milik Pop doang. Gate-nya sengaja inline `abort_unless(Auth::user()->isSuperadmin(), 403)` (bukan lewat Policy — endpoint ini dipakai lintas modul, jadi tidak terikat ke satu model Policy tertentu).
- **Authorization**: `App\Policies\ServicePolicy` — sama seperti policy modul lain, seluruh 5 method cuma `return $user->isSuperadmin();`.
- **Service layer**: `app/Services/ServiceService.php` — `create()` generate `pin` sebelum insert (independen dari id, tidak seperti `code` yang butuh id) dan `code` setelah insert; `update()`/`delete()` (soft).
- **Sidebar**: placeholder `'Layanan'` di `sidebar.blade.php` jadi link sungguhan ke `services.index`.
- Test: `tests/Feature/ServiceManagementTest.php` — CRUD superadmin, `code`/`pin` auto-generate (format 6 digit), `pin` bisa direset manual saat update dengan validasi `digits:6`, `user_id` ditolak kalau bukan role `customer`, `package_id` ditolak kalau paketnya bukan `is_starter`, `subdistrict_id`/`coverage_id`/`package_id` harus valid, `restrictOnDelete` mencegah hapus coverage yang masih dipakai service, soft delete (`assertSoftDeleted`, hilang dari listing default tapi row tetap ada), endpoint search customer ter-scope ke role `customer`, role selain superadmin dapat 403.

### Sales

Modul keenam yang sudah nyata — transaksi/order (`/sales`): mencatat produk apa saja yang dipesan untuk sebuah `service`, CRUD admin lewat Blade. Prasyarat (`services`) sudah lengkap sebelum modul ini dibangun; ini jadi bottleneck yang membuka Installation & Billing berikutnya (roadmap README menaruh `Installation → Billing → Invoice → Payment → Xendit` persis setelah `Service`). Scope iterasi ini **murni CRUD data transaksi** — TANPA integrasi Xendit/invoice/pembayaran (gap terpisah, `receipts` masih butuh sesi desain soal kolom spesifik Xendit sebelum ditulis jadi migration, lihat "Gap yang masih perlu didiskusikan" di bawah).

- **`starter`** (nama kolom di draft ERD) → **di-rename `is_starter`** di migration, konsisten `packages.is_starter`. Kolom baru (tidak ada data lama yang bergantung ke nama lama), jadi tidak ada alasan menyimpang penamaan dari konvensi yang sudah ada. Nilainya **snapshot** dari `packages.is_starter` milik `package_id` yang sedang dipilih — **dihitung ulang di setiap `create()` maupun `update()`** (bukan dibekukan sejak insert pertama), supaya tidak stale kalau staff ganti `package_id` saat edit sale.
- **`total`** = SUM(`sale_products.price × quantity`) semua baris, **sebelum** diskon apa pun. **`subtotal`** = `total − discount`. **`grandtotal`** = `subtotal + tax + admin_fee`. Rumus ini dikonfirmasi eksplisit ke user sebelum ditulis (draft ERD punya dua kolom harga mirip — `total` dan `subtotal` — yang maknanya tidak dijelaskan di diagram).
- **`sales.discount`** = SUM(`sale_products.discount`) — **turunan otomatis server-side** dari seluruh baris produk (`SaleService::syncProductsAndRecalculate()`), **bukan** input manual staff terpisah, supaya tidak ada dua sumber kebenaran yang bisa tidak sinkron. `tax` dan `admin_fee` tetap input manual staff (nullable, default 0) — belum ada rate/config pajak yang bisa dihitung otomatis saat ini.
- **`sales.package_id`** independen dari `services.package_id`. `services.package_id` tetap mencatat paket pendaftaran awal (tidak berubah, sudah berjalan sejak modul Service). `sales.package_id` mencatat paket APA yang dipesan di transaksi itu spesifik — **tidak harus selalu sama** dengan `services.package_id` saat ini, mendukung skenario upgrade/renewal di masa depan (belum ada UI/logic upgrade eksplisit di iterasi ini, sekadar field yang sudah siap). Konsekuensinya: `<select>` `package_id` di form Sales menampilkan **SEMUA** paket (starter maupun bukan) — beda dari form Service yang membatasi cuma paket starter (lihat "Service" di atas).
- **Line item produk dipilih individual dari katalog `Product`** (bukan auto-generate dari isi `package_product`) — staff pilih produk satu-satu lewat picker dinamis Alpine.js di `sales/_form.blade.php`, replikasi pola `packages/_form.blade.php` (`rows`/`addRow()`/`removeRow()`/`applyDefaultPrice()`), tapi direstruktur jadi `<table>` dengan 5 kolom (produk/qty/harga/diskon/satuan — dua kolom terakhir baru, tidak ada di form Package) karena terlalu banyak untuk ditampilkan sebagai flex row tanpa label. Guard produk duplikat dalam satu submission pakai pola sama persis `PackageRequest::guardDuplicateProducts()`.
- **`total`/`subtotal`/`discount`/`grandtotal` tidak pernah diterima dari request** — field ini sengaja **tidak ada** di `SaleRequest::rules()` sama sekali, jadi `$request->validated()` tidak pernah menyertakannya walau form/klien mengirimnya. Nilai final selalu dihitung ulang penuh di `SaleService::syncProductsAndRecalculate()` dari `products[]` yang disubmit — prinsip "jangan percaya angka dari client". Preview total di form (`sales/_form.blade.php`) dihitung live via Alpine `x-text`/getter murni untuk kenyamanan input, tapi elemen preview itu **tidak punya `<input name="...">`** sama sekali sehingga tidak pernah ikut submit.
- **`service_id`** dipilih lewat **AJAX typeahead**, replikasi persis pola picker customer yang baru dibangun di form Service (`serviceQuery`/`serviceId`/`fetchServices`/`searchServices`/`openServiceBrowse`/`selectService`, debounce 300ms, ambang mengetik minimal 3 karakter, `@focus` pada kolom kosong langsung browse tanpa menunggu ketikan). Endpoint `GET /sales/services/search` (`SaleController::searchServices()`, digate `$this->authorize('viewAny', Sale::class)`) cari di `code`/`address`/nama atau nomor telepon pelanggan terkait (`whereHas('user', ...)`), limit 20, query kosong tetap kembalikan daftar browse (bukan array kosong). **Tidak perlu filter manual soft-delete di query builder ini** — `Service::query()` otomatis exclude row yang soft-deleted lewat global scope `SoftDeletes`, selama tidak memanggil `withTrashed()`. **Beda dengan `SaleRequest::rules()`**: `Rule::exists('services', 'id')->whereNull('deleted_at')` di sana **wajib eksplisit**, karena `exists` rule Laravel query langsung ke tabel dan tidak menghormati global scope Eloquent — tanpa filter eksplisit itu, staff bisa submit `service_id` yang sudah soft-deleted lewat request langsung (bypass picker) dan tetap lolos validasi.
- **`service_id`/`package_id` pakai `restrictOnDelete()`** (konsisten pola `coverage_id`/`pop_id` di `services`). Catatan: `services` sudah pakai `SoftDeletes`, jadi constraint di `sales.service_id` baru "bergigi" kalau ada `forceDelete()` — tetap dipasang sebagai lapisan pertahanan konsisten konvensi project, bukan sia-sia (test-nya sengaja pakai `$service->forceDelete()` langsung di level model untuk memicu ini, karena `DELETE /services/{id}` normal cuma soft-delete dan tidak pernah menyentuh FK).
- **`sale_products`** (pivot `sales`↔`products`, dikelola via `Sale::products(): BelongsToMany` + `withPivot(['price','discount','quantity','unit'])`, sama pola `Package::products()`) punya kolom tambahan `price`/`unit` (snapshot dari `products.price`/`products.unit` saat baris ditambahkan, independen dari katalog yang bisa berubah setelahnya — pola sama `package_product.price`) dan `discount` (diskon per-line, sumber `sales.discount` di atas). Unique constraint `['sale_id', 'product_id']` — satu produk cuma boleh muncul sekali per sale (pola sama `package_product`), naikkan `quantity` kalau butuh lebih dari 1 unit.
- **`invoiced_at`/`expired_at`/`settled_at`/`canceled_at`** — kolom nullable timestamp biasa, **belum ada tombol/aksi transisi state machine** di iterasi ini (belum ada UI invoice/settle/cancel) — pola sama persis `activated_at`/`dismantled_at` di Service. Skema disiapkan duluan, logic-nya menyusul begitu modul Billing/Xendit dibangun.
- **`deleted_at` pakai `SoftDeletes` sungguhan** — Sales = riwayat transaksi, alasan sama seperti Service (beda dari master data katalog/infrastruktur yang hard delete).
- **Tidak menambah relasi balik** (`Service::sales()`/`Package::sales()`) di iterasi ini — tidak ada UI/kebutuhan yang memakainya sekarang (mis. belum ada halaman "riwayat transaksi" di detail Service). Tambahkan nanti kalau benar-benar dibutuhkan, jangan didesain lebih dulu untuk kebutuhan hipotetis.
- **Authorization**: `App\Policies\SalePolicy` — sama seperti policy modul lain, seluruh 5 method cuma `return $user->isSuperadmin();`.
- **Service layer**: `app/Services/SaleService.php` — `create()`/`update()` isi `is_starter`/`code`(setelah insert)/audit trail, delegasikan hitung total & sync pivot ke `syncProductsAndRecalculate()` (method **publik**, bukan `private`, supaya bisa dipanggil ulang dari `SaleSeeder` tanpa `Auth::id()` yang `null` di konteks console — method ini sama sekali tidak menyentuh kolom audit); `delete()` (soft).
- **Sidebar**: item baru `'Penjualan'` di `sidebar.blade.php`, ditaruh setelah `'Layanan'` dan sebelum placeholder `'Billing'` (Billing/invoice/Xendit tetap modul terpisah yang menyusul nanti, bukan direuse untuk Sales).
- **Seeder**: `database/seeders/SaleSeeder.php` — pakai `Sale::factory()->create([...])` langsung (bukan lewat `SaleService::create()`, karena `Auth::id()` `null` di konteks console) lalu panggil `app(SaleService::class)->syncProductsAndRecalculate($sale, $lines)` supaya total demo data tetap konsisten dengan rumus asli, bukan angka acak.
- Test: `tests/Feature/SaleManagementTest.php` — CRUD superadmin, `code` auto-generate, total/discount/subtotal/grandtotal terhitung benar dari line item (termasuk diskon per-line), `is_starter` snapshot dari package terpilih dan ikut berubah saat update, guard produk duplikat, `service_id` soft-deleted ditolak validasi, `service_id`/`package_id` harus valid, total tidak bisa dioverride dari request, update recalculate total saat line item berubah, soft delete, `restrictOnDelete` (termasuk kasus force-delete Service), endpoint search service (browse-on-empty, match code/address/nama/telepon, exclude soft-deleted), listing & search, render create/edit, role selain superadmin dapat 403.

## Draft Desain Database (`database/database_design.drawio`)

`database/database_design.drawio` adalah diagram ER draw.io — sebuah **draft roadmap** untuk skema MySQL, bukan spek yang final. Terbuka untuk direvisi; begitu migration sungguhan sudah ada untuk sebuah tabel, migration itulah yang jadi acuan utama, dan diagram dianggap historis/aspirasional untuk yang belum dibangun. Berdasarkan review terakhir, diagram ini mendefinisikan:

- `users` — akun (customer dan/atau staff, dibedakan lewat flag `admin`), dengan phone/code, password, `last_login_at`. Data KTP/dob/gender **tidak** lagi di kolom `users` langsung di draft ini — sudah dipisah ke tabel `user_details` (lihat poin berikutnya). **Implementasi sungguhan sudah menyimpang dari flag `admin` di draft ini** — diganti role lewat `spatie/laravel-permission` (lihat "Authorization / Role & Permission" di atas), kolom `admin` sudah di-drop dari `users`.
- `user_details` — tabel ekstensi 1:1 ke `users` (draft ini tidak menggambar FK eksplisit di kolomnya, hanya relasi lewat panah; sudah diimplementasikan sebagai shared primary key, lihat "User" di atas), berisi `nik` (nama field ini di draft sudah berganti dari `ktp` sebelumnya — CLAUDE.md sempat basi soal ini, sudah disinkronkan), `dob`, `gender`. **Implementasi sungguhan sudah menyimpang lagi dari draft ini**: kolom `dob` diimplementasikan sebagai `birth_date`, dan ada kolom `ktp_photo` yang sama sekali tidak ada di draft — keduanya keputusan yang diambil langsung dari instruksi user tanpa lewat draft drawio dulu. Diagram desain belum di-update untuk mencerminkan ini (di luar cakupan file itu untuk disentuh setiap saat; `database_actual.drawio` tetap yang jadi cermin akurat).
- `subdistricts` — data referensi wilayah Indonesia (kecamatan/kota/provinsi, dengan nama yang didenormalisasi).
- `pops` — perangkat PoP jaringan, terhubung ke sebuah subdistrict.
- `coverages` — area cakupan layanan, terhubung ke sebuah PoP.
- `products`, `packages`, `package_product` — katalog; package membundel beberapa product lewat tabel pivot. **Implementasi sungguhan sudah menyimpang dari draft ini** — lihat "Product & Package" di atas untuk detail: `product_type_id` (FK ke tabel `product_types` yang tidak pernah digambar) diganti kolom `type` (string + `Rule::in`), `packages.starter` diimplementasikan sebagai `is_starter` (boolean), dan `created_by`/`updated_by` (tidak ada di draft) ditambahkan sebagai audit trail.
- `services` — instance layanan yang dilanggan customer (alamat, coverage, timestamp lifecycle: `activated_at`/`expired_at`/`dismantled_at`/`canceled_at`/`deleted_at`).
- `sales` — order terhadap sebuah service + package (rincian harga: subtotal/discount/tax/admin_fee/grandtotal; lifecycle: `invoiced_at`/`settled_at`/`expired_at`/`canceled_at`).
- `sale_products` — line item untuk sebuah sale.
- `service_activations` / `service_dismantles` — event instalasi/uninstalasi yang terkait sebuah sale.
- `service_tickets` — tiket dukungan terhadap sebuah service.
- `receipts` — kwitansi pembayaran terkait sebuah sale.

Relasi utama: `pops`→`coverages`→`services`; `subdistricts`→`pops`/`services`; `users`→`services`; `products`/`packages`→`package_product`→`sale_products`; `services`→`sales`→{`sale_products`, `service_activations`, `receipts`}; `service_activations`→`service_dismantles`.

Gap yang sudah diselesaikan: tabel OTP (`otp_codes`) sudah dibangun dari nol (tidak ada di diagram sama sekali) — lihat "Authentication / Login" di atas. `users.phone`/`admin`/`last_login_at`/`code` juga sudah dimigrasikan, dan `user_details` (`nik`/`birth_date`/`gender`/`ktp_photo`) sudah dibangun sebagai tabel ekstensi shared-PK — lihat "User" di atas untuk detail keputusan single-table vs tabel `customers` terpisah (sempat dicoba, lalu dilebur balik ke `users` karena scope-nya bukan customer-eksklusif). `subdistricts` juga sudah dibangun sesuai kolom di diagram (migration `2026_07_08_100002_create_subdistricts_table.php`, model `App\Models\Subdistrict`, seeder `database/seeders/SubdistrictSeeder.php` yang meng-eksekusi dump SQL siap-pakai `database/seeders/subdistricts.sql` — ~84 ribu baris data wilayah Kemendagri, diimpor lewat `DB::unprepared` sekali saat tabel masih kosong). Tabel ini murni data referensi read-only (lookup wilayah provinsi/kota/kecamatan/kelurahan-desa yang didenormalisasi, kolom `district_id`/`city_id`/`province_id` adalah kode wilayah pemerintah, bukan FK — tidak ada tabel `districts`/`cities`/`provinces` terpisah) — sekarang sudah ada FK sungguhan _ke_ tabel ini dari `pops.subdistrict_id` (lihat "PoP & Coverage" di atas), meski belum ada controller/route/UI CRUD untuk `subdistricts` sendiri (dan memang tidak butuh, sifatnya read-only reference data yang di-seed sekali). `products`, `packages`, dan `package_product` juga sudah dibangun (dengan penyimpangan dari draft yang dicatat di poin sebelumnya) — lihat "Product & Package" di atas. `pops` dan `coverages` juga sudah dibangun sesuai kolom di diagram — lihat "PoP & Coverage" di atas untuk detail (termasuk `created_by`/`updated_by` yang ditambahkan sebagai audit trail, tidak ada di draft ini). `services` juga sudah dibangun — lihat "Service" di atas untuk detail penyimpangan dari draft (`pin` adalah password PPPoE bukan PIN verifikasi CS, `residential` di-rename `residential_name` karena isinya nama komplek bukan boolean, `deleted_at` pakai `SoftDeletes` sungguhan bukan kolom data biasa — satu-satunya modul yang soft delete sejauh ini, dan `package_id` ditambahkan langsung ke `services` — bukan menunggu tabel `sales` seperti relasi di draft ERD — supaya paket starter bisa dipilih langsung saat pendaftaran). `sales` dan `sale_products` juga sudah dibangun — lihat "Sales" di atas untuk detail penyimpangan dari draft (`starter` di-rename `is_starter` konsisten `packages.is_starter`, `discount` level-sale jadi turunan otomatis SUM dari `sale_products.discount` bukan input manual, `sales.package_id` independen dari `services.package_id` untuk mendukung transaksi susulan/upgrade, dan `deleted_at` pakai `SoftDeletes` sungguhan sama seperti Service).

Ada file drawio kedua, `database/database_actual.drawio`, dengan tujuan berbeda dari diagram desain di atas: itu bukan draft, melainkan cermin skema yang **sungguhan** sudah dimigrasikan ke database saat ini (`users`, `otp_codes`, `subdistricts`, `user_details`, `products`, `packages`, `package_product`, `pops`, `coverages`, `services`, `sales`, `sale_products`, plus tabel `spatie/laravel-permission`: `roles`, `permissions`, `model_has_roles`, `model_has_permissions`, `role_has_permissions`), kolomnya diambil langsung dari migration — bukan dari diagram desain, karena keduanya bisa saja menyimpang. Wajib di-update tiap kali ada migration baru landing, sama seperti kewajiban update CLAUDE.md ini sendiri.

Gap yang masih perlu didiskusikan dengan user sebelum ditulis jadi migration: `receipts` belum punya kolom spesifik Xendit (payment_request_id/method/status/paid_at/raw response) padahal Xendit adalah fokus utama.

### Dark mode

- Dark mode ikut preferensi sistem (`prefers-color-scheme`) secara default, tapi user bisa override manual lewat tombol toggle — permintaan eksplisit user supaya tidak dipaksa satu mode saja.
- Tailwind v4 dikonfigurasi pakai varian `dark:` berbasis **class**, bukan cuma media query: `@custom-variant dark (&:where(.dark, .dark *));` di `resources/css/app.css`. Ini wajib ada supaya toggle manual (menambah/menghapus class `dark` di `<html>`) benar-benar mengubah tampilan.
- `resources/views/components/theme-init.blade.php` — script kecil yang **harus** dirender paling awal di `<head>` (sebelum elemen lain), tugasnya membaca `localStorage.theme` (kalau user pernah memilih manual) atau fallback ke `matchMedia('(prefers-color-scheme: dark)')`, lalu toggle class `dark` di `<html>` sebelum browser sempat menggambar (mencegah flash tema salah). Juga listen ke perubahan preferensi sistem selama user belum override manual.
- `resources/views/components/theme-toggle.blade.php` — tombol Alpine yang toggle class `dark` + simpan pilihan ke `localStorage.theme` ('light'/'dark'). Dipasang di `header.blade.php` (untuk halaman ber-sidebar) dan pojok kanan-atas `auth-layout.blade.php` (halaman login/OTP, karena belum ada header di sana).
- Setiap komponen/halaman baru yang punya warna latar/teks eksplisit **wajib** ditambahi pasangan `dark:` yang sepadan (pola: `bg-white` → `+ dark:bg-gray-800`, `text-gray-900` → `+ dark:text-white`, `border-gray-300` → `+ dark:border-gray-700`, dst.) — jangan biarkan halaman baru cuma punya tampilan light mode. Sidebar (`sidebar.blade.php`) terkecuali: dia memang didesain gelap permanen ("dark-sidebar" ala Metronic) terlepas dari mode terang/gelap keseluruhan app.

### Logo mengikuti tema (light/dark)

`public/images/logo/` berisi dua varian logo (aset sementara dari `~/Webapp/xnet/app11`), keduanya PNG transparan berisi mark "X" saja (bukan logo dengan warna latar solid, meski namanya menyiratkan begitu):

- `logo-white-bg.png` — X berwarna **hitam**, didesain untuk dipasang di atas latar **terang**.
- `logo-black-bg.png` — X berwarna **putih**, didesain untuk dipasang di atas latar **gelap**.

Aturan pemakaian (dicontek dari pola yang sama di `~/Webapp/xnet/app/`, yang pakai trik CSS `dark:invert` pada satu logo hitam — di NEXA dipakai pendekatan dua file dengan toggle visibility karena kita sudah punya kedua varian siap pakai, bukan filter invert):

- **Area yang latarnya ikut berubah terang/gelap** (mis. `auth-layout.blade.php`, tempat logo duduk di atas `bg-gray-100 dark:bg-gray-900`) — render **kedua** `<img>`, satu dengan class `dark:hidden` (pakai `logo-white-bg.png`, utk light mode) dan satu lagi `hidden dark:block` (pakai `logo-black-bg.png`, utk dark mode).
