## Status Project

Ini adalah **NEXA**, aplikasi internal untuk XNet (PT. XPlus Network Indonesia) yang akan menjadi tulang punggung administrasi/operasional untuk perusahaan ISP: manajemen pelanggan, katalog produk/layanan, billing, dan pembayaran via Xendit, yang selanjutnya akan berkembang ke integrasi jaringan (MikroTik, OLT) dan operasional ISP secara lebih luas.

Repository ini masih tahap awal. Modul domain yang sudah nyata (bukan sekadar shell) adalah **Authentication** (login phone + OTP WhatsApp), **User** (CRUD admin untuk seluruh akun — customer, staff, teknisi, admin, bukan cuma customer), **Product & Package** (katalog produk + paket bundling), **PoP & Coverage** (master data infrastruktur jaringan — titik PoP + area cakupan layanan), **Service** (instance langganan pelanggan — alamat pemasangan + koneksi ke coverage), **Sales** (transaksi/order produk terhadap sebuah service), **Notification** (fondasi in-app + WhatsApp + email, belum ada trigger bisnis nyata), dan **Billing / Invoice (Xendit)** (tagihan pendaftaran otomatis, Payment Requests API v3, webhook, auto-cancel terjadwal) — lihat "Authentication / Login", "User", "Product & Package", "PoP & Coverage", "Service", "Sales", "Notification", dan "Billing / Invoice (Xendit)" di bawah untuk detail implementasi. Modul bisnis lain (Installation, Renewal/reminder, Dismantle, dst.) belum dibangun. `README.md` adalah dokumen perencanaan arsitektur yang hidup (living document, dalam Bahasa Indonesia) yang menjelaskan arah project; jadikan itu sebagai sumber kebenaran untuk struktur dan konvensi yang dituju, sampai kode yang benar-benar berjalan menetapkan sebaliknya.

## Arah Arsitektur (dari README.md)

Project ini bermaksud mengikuti konvensi berikut saat kode domain mulai ditambahkan — terapkan ini saat membuat fitur baru, jangan default ke controller gemuk:

- **Layering**: Form Request untuk validasi, Service Layer untuk business logic, Eloquent untuk persistence. Jauhkan business logic dari Controller; gunakan dependency injection.
- **Konvensi penamaan** per konsep domain, contoh untuk entitas `Customer`: `CustomerController`, `CustomerService`, `CustomerPolicy`, `CustomerRequest`.
- **API**: aplikasi customer-facing (web/Android/iOS) akan mengakses lewat REST API versioned di `/api/v1`, terpisah dari admin UI berbasis Blade.
- **Pembagian teknologi frontend** — gunakan teknologi sesuai tanggung jawabnya, jangan default pakai Vue:
    - Blade: layout, dashboard, halaman CRUD, form, table.
    - Alpine.js: interaksi ringan (modal, dropdown, accordion, toast, toggle).
    - Vue: hanya untuk komponen yang benar-benar kompleks (dashboard realtime, monitoring jaringan, grafik, wizard multi-step).
- **Integrasi jaringan**: integrasi MikroTik dan OLT (HSGQ) direncanakan di belakang pola adapter/driver supaya vendor baru bisa ditambahkan tanpa menyentuh modul bisnis.
- Soft delete hanya jika benar-benar dibutuhkan, bukan default; timestamp standar Laravel; foreign key dan index yang sesuai diharapkan ada di semua skema.

### Authentication / Login

Sudah **diimplementasikan sungguhan** (bukan lagi shell/stub) — login pakai nomor telepon + OTP WhatsApp, lengkap dengan migration, service layer, dan middleware auth.

- Login menggunakan **nomor telepon**, bukan email/password — berbasis OTP, bukan login password tradisional untuk end user.
- OTP dikirim melalui **WhatsApp**, bukan SMS.
- WhatsApp gateway adalah **service terpisah** (`go-whatsapp-web-multidevice`), diakses lewat REST API. NEXA tidak berkomunikasi langsung dengan WhatsApp/Meta.
- NEXA sendiri yang memegang lifecycle OTP — pembuatan, penyimpanan, masa berlaku, dan validasi semuanya terjadi di aplikasi ini, bukan di service WA gateway.
- **Tidak ada auto-register.** Nomor yang belum ada di tabel `users` ditolak saat minta OTP (`SendOtpRequest`, rule `exists:users,phone`) — pembuatan akun customer adalah tanggung jawab modul Customer yang belum dibangun. Untuk dev lokal, `database/seeders/AdminUserSeeder.php` menyediakan satu akun admin (`phone` `6281234567890`) via `php artisan migrate:fresh --seed`.
- **NEXA khusus admin/staff — role `customer` sengaja ditolak login**, ditegakkan **dua lapis**: (1) `SendOtpRequest::withValidator()` menolak dengan error field `phone` sebelum OTP sempat dikirim kalau nomor terdaftar sebagai customer (`$user->isCustomer()`, convenience method baru di `User` model — pola sama `isSuperadmin()`, jangan sebar `hasRole('customer')` literal di banyak file); (2) `LoginController::verifyOtp()` cek ulang tepat sebelum `Auth::login()`, jaga-jaga kalau role user berubah di antara request OTP dan verifikasi. Alasan: aplikasi customer-facing terpisah (web/Android/iOS, lihat README "API") belum dibangun — sebelum guard ini ada, customer bisa login ke NEXA (dashboard admin) begitu saja karena `SendOtpRequest` cuma cek "terdaftar", bukan "role apa".
- **Format nomor telepon: selalu disimpan dalam format internasional (kode negara `62` + nomor, tanpa `0` di depan), kolom `phone` bertipe `unsignedBigInteger` (BIGINT UNSIGNED) — bukan string/VARCHAR.** Keputusan sadar: nomor Indonesia dianggap selalu diawali kode negara 62 begitu dinormalisasi, jadi tidak ada leading-zero bermakna yang perlu dijaga (leading `0` yang ada di input user hanyalah trunk prefix domestik yang dibuang, bukan bagian dari nomor kanonis). Field login di `resources/views/auth/login.blade.php` tetap **satu input polos** (placeholder `81234567890`, tanpa badge/prefix visual `+62`) — user cukup ketik nomor lokal tanpa `0`/`62`, backend yang menormalisasi. Normalisasi dilakukan oleh `App\Support\PhoneNumber::normalize()` (dipanggil dari `SendOtpRequest::prepareForValidation()`): input dibersihkan dari karakter non-digit, lalu kalau diawali `0` diganti jadi `62`, kalau sudah diawali `62` dibiarkan, selain itu `62` ditambahkan di depan. Pakai helper ini lagi (jangan re-implement) di modul lain yang menerima input nomor telepon (mis. nanti Customer). `User::phone` di-cast `'integer'` di model.
- **Skema**: migration `..._add_phone_admin_last_login_at_to_users_table` menambah `phone` (unsignedBigInteger, nullable, unique — nullable supaya tidak konflik dengan baris user yang belum punya phone; "harus terdaftar" ditegakkan di Form Request, bukan di DB), `admin` (boolean), `last_login_at` ke `users`. Kolom domain customer dari draft drawio (`ktp`/`dob`/`gender`/`code`) **sengaja ditunda** ke migration terpisah saat modul Customer dikerjakan. Tabel baru `otp_codes` (`app/Models/OtpCode.php`) — `user_id` (FK, bukan kolom `phone` terpisah, karena OTP hanya bisa diminta untuk user yang sudah ada), `code_hash` (HMAC-SHA256, bukan bcrypt — OTP berumur pendek/sekali pakai dan sudah dilindungi rate limit + attempt lockout), `expires_at`, `consumed_at`, `attempts`.
- **Service layer**: `app/Services/OtpService.php` (generate/simpan/expire/validasi kode, cooldown resend 60 detik via `Cache`, lockout setelah `OTP_MAX_ATTEMPTS` percobaan salah), dipanggil dari `LoginController` (sudah bukan stub lagi, di-inject `OtpService`).
- **WhatsApp gateway pakai pola driver/adapter** (`app/Services/Whatsapp/`): interface `WhatsappGateway`, driver **`log`** (`LogWhatsappGateway` — tulis OTP ke `storage/logs/laravel.log`, dipakai kalau belum ada server gateway yang bisa diakses) dan driver **`http`** (`HttpWhatsappGateway`, raw `Http::` facade sesuai mandat README) untuk server `go-whatsapp-web-multidevice` sungguhan. Pilihan driver dari `config('services.whatsapp.driver')` / env `WHATSAPP_GATEWAY_DRIVER`, di-bind di `app/Providers/WhatsappServiceProvider.php` (terdaftar di `bootstrap/providers.php`). **Kredensial gateway sungguhan sudah diisi di dev** (`WHATSAPP_GATEWAY_DRIVER=http`) — beda dari kebanyakan REST API, server ini diautentikasi lewat **HTTP Basic Auth** (`WA_GATEWAY_URL`/`WA_GATEWAY_USERNAME`/`WA_GATEWAY_PASSWORD`, dipetakan ke `config('services.whatsapp.url'/'username'/'password')`, prefix env sengaja beda dari `WHATSAPP_GATEWAY_DRIVER`/`WHATSAPP_GATEWAY_LOG_CHANNEL` yang lama — ikut penamaan yang sudah dipakai user untuk kredensial gateway ini), bukan Bearer token seperti dugaan awal — `HttpWhatsappGateway` memakai `->withBasicAuth($username, $password)`. Endpoint path/payload (`POST /send/message`, body `{phone, message}`) masih **best-effort** (belum ada dokumentasi resmi yang dirujuk), revisit begitu diuji end-to-end. **Test suite selalu memaksa `WHATSAPP_GATEWAY_DRIVER=log`** lewat `phpunit.xml` (pola sama `MAIL_MAILER=array`) — supaya test tidak pernah bergantung pada `.env` developer atau memanggil gateway sungguhan lewat jaringan.
- **Pengiriman OTP lewat job `app/Jobs/SendOtpWhatsappNotification.php`, tapi di-dispatch di koneksi queue `sync`** (`config('otp.queue_connection')` / env `OTP_QUEUE_CONNECTION`, default `sync`), **bukan ikut `QUEUE_CONNECTION=database` global**. Ini keputusan sadar, bukan default Laravel: percobaan pertama pakai `database` sempat menyebabkan job nyangkut di tabel `jobs` tanpa pernah diproses karena tidak ada `queue:work`/`queue:listen` yang jalan di dev — OTP jadi tidak pernah benar-benar "terkirim" (tidak ke-log), padahal baris OTP di DB sudah dibuat, bikin bingung ("kode di log kok tidak cocok" — padahal log-nya memang belum pernah ditulis, itu log basi dari percobaan sebelumnya). `sync` membuat job jalan inline saat request tanpa butuh worker terpisah. Kalau nanti `HttpWhatsappGateway` sungguhan dipakai di production dan panggilan HTTP-nya berpotensi lambat/nge-block request, pertimbangkan ubah `OTP_QUEUE_CONNECTION` kembali ke `database` (atau koneksi queue khusus) — jangan asumsikan `sync` cocok selamanya.
- **Routing & middleware**: `/login` + `/login/otp` dalam grup `guest`; `/dashboard` + `POST /logout` (baru) dalam grup `auth`. `bootstrap/app.php` pakai `redirectGuestsTo('/login')`/`redirectUsersTo('/dashboard')` (built-in Laravel 13). Rate limiter `otp-request` (3/menit per IP+phone) dan `otp-verify` (10/menit per IP) didaftarkan di `AppServiceProvider::boot()` — karena tidak ada `RouteServiceProvider` terpisah di skeleton Laravel 13 ini.
- Tombol "Keluar" di `header.blade.php` sudah di-wire ke `POST /logout` (form + `@csrf`), menggantikan placeholder `<a href="#">`.
- Test: `tests/Feature/Auth/LoginFlowTest.php` — pakai fake `WhatsappGateway` (bind instance yang menangkap kode asli) untuk membaca kode OTP tanpa bergantung pada skema hash internal. Termasuk guard customer: request OTP ditolak & tidak ada OTP terkirim, dan verifikasi tetap ditolak walau sesi OTP sudah ada (role berubah jadi customer di antara request dan verify).

### User (dulu "Customer" — scope sengaja dilebarkan)

Modul kedua yang sudah nyata — CRUD admin lewat Blade di **`/users`** (list/create/edit/delete), belum ada endpoint `/api/v1` (menyusul saat customer app sungguhan mulai dibangun).

Awalnya dibangun sebagai modul "Customer" murni (`/customers`, model `Customer extends User` dengan global scope `admin=false`). **User lalu eksplisit meralat**: layar ini bukan cuma untuk kelola customer, tapi seluruh akun (customer, staff, teknisi, admin) — jadi model `Customer` **dihapus total**, diganti operasi langsung ke `App\Models\User`, dan route/URL pindah ke `/users`. Konvensi ke depan: **default ke istilah "user"/"pengguna", bukan "customer"/"pelanggan"**, kecuali fitur itu memang eksklusif milik customer (mis. nanti billing/invoicing yang nempel ke `sales`).

- **Satu tabel `users` untuk semua role**, dibedakan lewat role dari `spatie/laravel-permission` — lihat section "Authorization / Role & Permission" di bawah untuk detail. Tidak ada lagi kolom `users.admin` (sudah di-drop) maupun subclass `Customer` — Controller/Service/Request/Policy semua bertipe `App\Models\User` langsung.
- **Pilihan role** ada di form create/edit (`resources/views/users/_form.blade.php`, `<select name="role">`) — superadmin yang login bisa membuat/menandai role akun lain lewat UI yang sama, bukan cuma lewat `AdminUserSeeder`. **Guard yang sudah ada**: superadmin tidak bisa mengubah role akun miliknya sendiri keluar dari `superadmin` (`UserRequest::guardAgainstSelfDemotion()`) — supaya tidak keunci sendiri dari sistem selama belum ada mekanisme recovery lain. Guard granular lain (mis. mencegah superadmin biasa mengelola superadmin lain) **belum diimplementasikan** — user sudah bilang eksplisit itu "dipikirkan lagi nanti" sesuai kebutuhan; jangan improvisasi skema permission granular sebelum diminta.
- **`users.code`** hanya digenerate untuk role **`customer`** (`'CUS'.str_pad($user->id, 6, '0', STR_PAD_LEFT)` di `UserService::create()`, setelah insert — aman dari race condition karena mengandalkan auto-increment id) — role lain (`superadmin`/`technician`/`finance`/`sales`) dibiarkan `code = null` karena `code` itu untuk keperluan invoicing/billing customer (lihat draft ERD), bukan staff.
- **`user_details`** — tabel ekstensi 1:1 dengan `users`, **shared primary key** (`user_details.id` = `users.id`, FK cascade delete, bukan auto-increment sendiri), **berlaku untuk semua role** (bukan customer-eksklusif — makanya relasi `userDetails()` didefinisikan di `User`). Kolom: `nik` (string 16 char, bukan `unsignedBigInteger` seperti `phone` — alasan normalisasi leading-zero yang melatarbelakangi keputusan `phone` tidak berlaku untuk NIK, diperlakukan sebagai identifier opak), `birth_date` (date), `gender` (string, nilai `'male'`/`'female'`), `ktp_photo` (string path — lihat "Foto KTP" di bawah untuk lokasi/akses). Model `App\Models\UserDetail` (`$incrementing = false`).
- **NIK bukan input bebas** — `gender` dan `birth_date` **selalu** diturunkan dari NIK lewat `UserDetail::parseNik(string $nik): ?array`, tidak pernah diinput manual. Struktur NIK: digit 7-8 = hari lahir (kalau > 40, kurangi 40 dan tandai `gender = 'female'`; kalau tidak, `'male'`), digit 9-10 = bulan, digit 11-12 = tahun 2 digit — abad ditentukan dengan mencoba `[abad sekarang, abad sekarang - 100]` dan pilih yang menghasilkan usia masuk akal (17–100 tahun relatif ke `now()`, dicek pakai `Carbon::diffInYears`). `checkdate()` PHP dipakai untuk validasi hari/bulan/tahun kabisat. Return `null` kalau tidak 16 digit, bulan/hari tidak valid, atau tidak ada kandidat abad yang menghasilkan usia 17–100 tahun — `null` berarti NIK ditolak di `UserRequest` (`UserDetail::parseNik($nik) === null` → error validasi "NIK tidak valid").
- **NIK dikunci setelah pertama kali tersimpan** — begitu `user_details.nik` sudah terisi, tidak bisa diubah lewat form manapun (termasuk form superadmin), ditegakkan di `UserRequest::guardNik()` (bandingkan NIK yang datang vs `$target->userDetails->nik` yang sudah ada; beda → error "NIK sudah terkunci"). Belum ada mekanisme override khusus untuk ini; kalau perlu koreksi, sementara lewat `tinker`/DB langsung. Form edit tidak lagi merender `<input>` untuk NIK begitu sudah terkunci — cuma teks statis + `<input type="hidden">` supaya tetap ikut ter-submit tapi tidak bisa diketik ulang (pertahanan UI; validasi server tetap penegak utama — **wajib** render `@error('nik')` di luar percabangan terkunci/tidak, supaya percobaan tamper tetap dapat feedback, bukan gagal senyap).
- **Kolom `password` di `users` tetap NOT NULL** (peninggalan scaffolding default Laravel) tapi tidak pernah dipakai — login selalu lewat OTP. `UserService::create()` mengisi placeholder inert `Hash::make(Str::random(40))`, tidak pernah disentuh lagi setelahnya (tidak ada UI untuk mengubahnya).
- **`email` adalah field asli & wajib** (bukan placeholder lagi) — input eksplisit di form create/edit (`resources/views/users/_form.blade.php`), divalidasi `required|email:rfc|unique:users,email` di `UserRequest`. **Ini perubahan dari perilaku awal**: sebelumnya `UserService` mengisi placeholder inert `"user-{$phone}@nexa.internal"` karena login selalu OTP dan email dianggap tidak pernah dipakai — diubah karena channel email (Notification module) dan komunikasi ke user butuh alamat asli, dan staff/admin nyatanya punya email sungguhan yang wajar diminta saat pendaftaran. **Akun lama yang masih punya placeholder tidak otomatis dimigrasikan** — staff harus mengisi email asli lewat form edit kalau perlu (tidak ada migration data untuk retrofit ini).
- **Nama dinormalisasi ke Title Case** — `UserRequest::prepareForValidation()` mengubah tiap kata jadi diawali huruf besar lewat `App\Support\TitleCase::normalize()` (`Str::title()`, spasi ganda dirapikan) sebelum disimpan, mis. `"budi   santoso"` → `"Budi Santoso"`. Berlaku di create maupun update (field `name` sama-sama lewat `UserRequest`), jadi nama lama yang belum konsisten ikut ternormalisasi begitu diedit ulang. Helper ini awalnya bernama `PersonName`, **direname jadi `TitleCase`** begitu dipakai ulang untuk field non-nama-orang juga (`address`/`residential_name` di `ServiceRequest`, lihat "Service" di bawah) — nama class lama akan menyesatkan kalau tetap dipakai untuk alamat.
- **Notifikasi WhatsApp selamat datang** — `UserService::create()` mengirim `App\Notifications\UserRegisteredNotification` (`via()` = `['database', WhatsappChannel::class]`, tanpa `mail`) lewat `NotificationService::send()` setelah akun berhasil dibuat, untuk **semua role**, bukan cuma customer. Pesannya generik (menyebut label role dalam Bahasa Indonesia, mis. "Pelanggan"/"Teknisi") — tidak menyebut instruksi login supaya tetap benar untuk role `customer` yang justru **tidak boleh** login ke NEXA (lihat "NEXA khusus admin/staff" di atas).
- **Authorization**: `App\Policies\UserPolicy` (viewAny/view/create/update/delete → `$user->isSuperadmin()`) — lihat section "Authorization / Role & Permission" untuk detail role. Laravel auto-discover Policy lewat konvensi nama (`User` model ↔ `UserPolicy`), dipanggil dari `UserController::__construct()` via `$this->authorizeResource(User::class, 'user')`. `App\Http\Controllers\Controller` (base class) di-extend dari `Illuminate\Routing\Controller` (`$this->middleware()`) + `use AuthorizesRequests` supaya `authorizeResource()` jalan.
- **Service layer**: `app/Services/UserService.php` — `create()`/`update()` jalan dalam `DB::transaction()`, `delete()` hard delete.
- **Request**: satu `App\Http\Requests\UserRequest` dipakai untuk create & update, reuse `PhoneNumber::normalize()` di `prepareForValidation()`.
- Sidebar "Pengguna" (`resources/views/components/sidebar.blade.php`) mengarah ke `users.index` (dulu "Pelanggan" → `customers.index`).
- Test: `tests/Feature/UserManagementTest.php` — CRUD superadmin, code auto-generate hanya utk role `customer`, listing gabungan lintas role, guard self-demotion role, validasi unique phone/nik/email, NIK lock, role selain superadmin dapat 403, akses foto KTP privat, normalisasi nama ke Title Case, notifikasi WhatsApp selamat datang terkirim ke akun baru (pakai `CapturingWhatsappGateway`, pola sama `LoginFlowTest`), dan endpoint `complete-kyc` (lihat "Service" — sukses, ditolak kalau NIK sudah pernah diisi, 403 non-superadmin).

### Authorization / Role & Permission

Dibangun setelah modul User, langsung menyelesaikan dua hal yang sengaja ditunda di sana: lock NIK belum punya override, dan CRUD `/users` cuma biner admin/bukan-admin.

- **Package**: `spatie/laravel-permission` (bukan hand-roll) — dipasang via `composer require` + `php artisan vendor:publish --provider="Spatie\Permission\PermissionServiceProvider"`, menerbitkan `config/permission.php` dan migration `create_permission_tables` (tabel `roles`, `permissions`, `model_has_roles`, `model_has_permissions`, `role_has_permissions`). Tidak pakai fitur teams (`permission.teams` tetap `false`, default) karena belum dibutuhkan.
- **Satu role per user** (bukan many-to-many) — cukup untuk kebutuhan saat ini, dipilih user secara sadar; package tetap mendukung upgrade ke multi-role nanti (`syncRoles()` dipakai bukan `assignRole()` doang di `update()`, supaya nanti gampang diperluas).
- **5 role**: `superadmin`, `technician`, `finance`, `sales`, `customer` — daftar final dari user, **bukan** set customer/staff/teknisi/admin yang sempat disebut di context `user_details` sebelumnya (itu istilah umum, bukan daftar role). Di-seed lewat `Role::firstOrCreate(['name' => ..., 'guard_name' => 'web'])`, baik di migration data (lihat poin berikutnya) maupun `database/seeders/RoleSeeder.php` (idempotent, didaftarkan di `DatabaseSeeder` **sebelum** `AdminUserSeeder` supaya `migrate:fresh --seed` di masa depan tetap benar).
- **Migrasi data `users.admin` → role, lalu drop kolom** — dua migration terpisah (urutan penting): `2026_07_09_005247_assign_roles_from_admin_flag.php` (baca `admin` boolean yang masih ada lewat `DB::table('users')`, assign role `superadmin` kalau `true` else `customer`, lalu **baru setelah itu** kolom di-drop) dan `2026_07_09_005247_drop_admin_from_users_table.php`. Sengaja dipisah dari schema-change supaya data-migration dan drop-column masing-masing punya `down()` yang jelas.
- **`User` model**: pakai trait `Spatie\Permission\Traits\HasRoles`, method `isSuperadmin(): bool { return $this->hasRole('superadmin'); }` — satu tempat convenience, dipakai di `UserPolicy` & `KtpPhotoController` (jangan sebar `hasRole('superadmin')` literal di banyak file, biar gampang diperluas kalau nanti perlu OR beberapa role).
- **Gate masih konservatif — cuma `superadmin`.** Permission granular per role (siapa boleh apa persis di modul mana) **belum didesain** — user eksplisit bilang itu dipikirkan sesuai kebutuhan nanti. Supaya tidak berimprovisasi bikin skema permission yang mengada-ada, iterasi ini cuma **menggeser** gate `isAdmin()` lama jadi `isSuperadmin()` — perilaku setara persis dengan sebelumnya (dulu cuma `admin=true` yang boleh akses `/users` & lihat KTP orang lain, sekarang cuma role `superadmin`). `technician`/`finance`/`sales` **belum** dapat akses apa pun ke `/users` — itu menyusul begitu kebutuhan per-role jelas. Jangan tambah permission granular tanpa diminta eksplisit.
- Belum ada tabel `permissions` yang benar-benar dipakai (infrastrukturnya ada dari package, tapi belum ada permission bernama yang di-assign ke role manapun) — dan belum ada UI untuk kelola role/permission itu sendiri (create role baru, assign permission ke role). Keduanya gap yang didokumentasikan, bukan lupa.

#### Foto KTP (private storage)

- Disimpan di disk `local` (`config/filesystems.php` sudah root ke `storage/app/private` secara default di Laravel 13 — bukan `storage/app/public`), subfolder `ktp/` (`$file->store('ktp', 'local')` di `UserService`). **Tidak** ada di disk `public`, tidak ke-link oleh `php artisan storage:link`, jadi tidak bisa diakses langsung lewat URL `/storage/...` sama sekali.
- Serving lewat `GET /secure/ktp/{user}` → `App\Http\Controllers\KtpPhotoController@show`, route name `secure.ktp`, terdaftar di grup middleware `auth`. Otorisasi inline (bukan Policy — cukup sederhana untuk 1 aksi): boleh diakses **pemilik akun sendiri** (`$viewer->id === $user->id`) **atau role `superadmin`** (`$viewer->isSuperadmin()`). Selain itu → 403. Kalau `ktp_photo` null → 404. Response pakai `Storage::disk('local')->response($path)` (inline, bukan `download()`, supaya `<img>` tag bisa render langsung).
- Di view, **selalu** pakai `route('secure.ktp', $user)`, jangan `asset('storage/...')` — pola lama (waktu masih disk `public`) sudah diganti di `resources/views/users/_form.blade.php`.

### Product & Package

Modul ketiga yang sudah nyata — katalog produk (`/products`) dan paket bundling produk (`/packages`), CRUD admin lewat Blade, prasyarat untuk modul Service/Billing/Sales yang akan datang. Struktur (Model/Policy/Request/Service/Controller) direplikasi persis dari pola modul User.

- **Penyimpangan sadar dari draft ERD**: `database/database_design.drawio` punya kolom `products.product_type_id` yang merujuk tabel `product_types` — tabel itu **tidak digambar sama sekali** di diagram (FK menggantung). Diputuskan **bukan** dibuatkan tabel lookup terpisah untuk iterasi ini; diganti kolom `products.type` (string), divalidasi `Rule::in` dengan daftar tetap: `perangkat`, `jasa`, `biaya`, `langganan`, `lainnya` (`App\Http\Requests\ProductRequest::TYPES`). Draft ERD belum diperbarui untuk mencerminkan ini — kalau nanti kebutuhan kategori produk berkembang (banyak atribut per kategori, dsb.), baru dipertimbangkan naik jadi tabel FK sungguhan.
- **`products.type = 'langganan'`** menandai produk berlangganan berulang (mis. "Internet 10 Mbps") — beda dari `perangkat`/`jasa`/`biaya`/`lainnya` yang sekali beli. **1 unit `quantity` produk bertipe ini di dalam sebuah `package_product` = 1 bulan masa aktif** ketika dibundel ke sebuah `Package` — lihat `packages.duration_months` di bawah. Ditambahkan khusus untuk mendukung alur registrasi→aktivasi→billing yang didiskusikan bareng user (durasi masa aktif service diturunkan dari sini), mengadopsi pola `Product::isSubscription()` di app sebelumnya (`~/Webapp/xnet/app`) tapi lewat tipe eksplisit di `Rule::in`, bukan kolom boolean terpisah.
- **`packages.duration_months`** (unsignedSmallInteger, default `1`) — kolom cache, dihitung ulang oleh `PackageService::deriveDurationMonths()` setiap `create()`/`update()` menyimpan produk paket: diambil dari `quantity` item bertipe `langganan` di paket tsb (**bukan** dijumlah kalau ada lebih dari satu item langganan — `PackageRequest::guardSubscriptionDuration()` mewajibkan quantity-nya seragam kalau lebih dari satu, error di field `products` kalau tidak). Default `1` kalau paket tidak punya item `langganan` sama sekali. Ini **beda pola** dari app sebelumnya (yang cuma ambil quantity item langganan **pertama**, tanpa guard keseragaman) — NEXA sengaja menegakkan validasi eksplisit supaya tidak ambigu durasi mana yang dipakai kalau staff keliru input.
- **`packages.is_starter`** (boolean, default `false`) — nama kolom diselaraskan ke konvensi Laravel dari `packages.starter` di draft ERD (yang tidak ada keterangan di diagram, dikonfirmasi maknanya langsung ke user). `true` berarti paket bisa dipilih saat **pendaftaran baru** (mis. promo gratis 3 bulan + gratis pasang + modem premium bundel jadi satu paket starter); `false` berarti paket hanya bisa dipilih pelanggan yang **sudah aktif** berlangganan (mis. paket upgrade/add-on). Belum ada logic yang menegakkan pembedaan ini di alur pendaftaran/langganan (menyusul begitu modul Service/Sales dibangun) — untuk sekarang kolom ini murni data, ditampilkan sebagai badge Ya/Tidak di `packages/index.blade.php`.
- **`created_by`/`updated_by`** (nullable, FK ke `users`, `nullOnDelete`) — **modul pertama yang mulai pola audit trail** di NEXA (User/OtpCode belum punya kolom ini). Diisi `Auth::id()` di `ProductService`/`PackageService`, bukan retrofit ke modul User yang sudah ada. Kalau pola ini terbukti berguna, pertimbangkan menambahkannya ke `users` lewat migration terpisah nanti — jangan diam-diam diseragamkan tanpa keputusan eksplisit.
- **`code`** (`PRD`/`PKG` + 6 digit id, `str_pad`) — **selalu** digenerate untuk setiap produk/paket (beda dari `users.code` yang cuma untuk role `customer`), karena setiap baris katalog memang butuh kode bisnis. Sama seperti `users.code`, kolom `code` di migration harus **nullable** (bukan cuma unique) — digenerate lewat `update()` setelah insert awal supaya bisa memakai id auto-increment, jadi insert pertama tidak menyertakan `code` sama sekali.
- **`package_product`** (pivot `packages`↔`products`) — punya kolom tambahan `quantity` dan `price` (snapshot harga produk saat dibundel ke paket, independen dari `products.price` yang bisa berubah setelahnya), plus unique constraint `['package_id', 'product_id']` (satu produk cuma boleh muncul sekali per paket — kalau butuh lebih dari 1 unit, naikkan `quantity`, bukan baris baru). Dikelola lewat relasi `BelongsToMany` biasa (`Package::products()` / `Product::packages()`, `withPivot(['quantity', 'price'])->withTimestamps()`) dan `sync()` di `PackageService` — tidak perlu pivot model kustom untuk kebutuhan ini.
- **Form paket** (`resources/views/packages/_form.blade.php`) punya picker produk dinamis: baris produk (pilih produk + quantity + harga) bisa ditambah/dihapus lewat Alpine.js `x-data` inline (array `rows`, method `addRow()`/`removeRow()`/`applyDefaultPrice()`) — konsisten dengan pembagian teknologi frontend di CLAUDE.md (Alpine untuk interaksi ringan, bukan Vue). `PackageRequest` menolak `product_id` duplikat dalam satu submission lewat `withValidator()` + guard method, pola yang sama seperti `UserRequest::guardNik()`.
- **Authorization**: `App\Policies\ProductPolicy` dan `App\Policies\PackagePolicy` — sama seperti `UserPolicy`, seluruh 5 method (`viewAny/view/create/update/delete`) cuma `return $user->isSuperadmin();`. Belum ada akses untuk role lain (`technician`/`finance`/`sales`/`customer`), konsisten dengan gate konservatif yang sudah ada di `/users` — permission granular per role masih ditunda sesuai kebutuhan nanti.
- **Service layer**: `app/Services/ProductService.php`, `app/Services/PackageService.php` — `create()`/`update()`/`delete()`, isi `code`/`created_by`/`updated_by`, dan (khusus Package) `sync()` pivot produk.
- **Sidebar**: item placeholder `'Produk & Paket'` di `sidebar.blade.php` dipecah jadi dua item nyata terpisah — `'Produk'` (→ `products.index`) dan `'Paket'` (→ `packages.index`) — karena keduanya dua resource CRUD terpisah, bukan satu modul gabungan. Belum ada infra submenu/accordion di sidebar untuk mengelompokkan keduanya di bawah satu label.
- Test: `tests/Feature/ProductManagementTest.php`, `tests/Feature/PackageManagementTest.php` — CRUD superadmin, validasi `type`, `code` auto-generate, bundling produk ke paket (quantity/price pivot), guard produk duplikat dalam satu paket, `is_starter` toggle, role selain superadmin dapat 403.
- **Seeder**: `database/seeders/ProductPackageSeeder.php` — 7 produk & 5 paket meniru **persis** nama/harga/bundling demo data app sebelumnya (`~/Webapp/xnet/app/database/seeders/{Product,Package}Seeder.php`), bukan `fake()` acak — supaya demo NEXA punya konteks bisnis ISP yang masuk akal (Internet Basic/Standar/Premium, Biaya Instalasi, Modem Standar/Premium, Mesh WiFi; paket starter 1 bulan, paket renewal semesteran `is_starter=false`, paket promo pendaftaran 3 bulan). Dibuat lewat Eloquent langsung (bukan `ProductService`/`PackageService`, karena `Auth::id()` `null` di konteks console — pola sama seperti `SaleSeeder`), `code` dan `packages.duration_months` dihitung manual meniru logic service layer asli.

### Service

Modul kelima yang sudah nyata — instance langganan pelanggan (`/services`): alamat pemasangan + koneksi ke `coverage` + `package` yang didaftarkan, CRUD admin lewat Blade. Prasyarat untuk Installation/Billing/Sales yang akan datang. Draft ERD (`database/database_design.drawio`) punya beberapa kolom yang maknanya sempat ambigu dan dikonfirmasi langsung ke user sebelum migration ditulis — jangan improvisasi ulang tebakan lama (PIN verifikasi CS, `residential` sebagai boolean) yang sempat salah duga di sesi perencanaan sebelum ini:

- **`pin`** — **bukan** PIN verifikasi CS, tapi **password PPPoE** (6 digit) yang dipakai autentikasi router pelanggan ke MikroTik. Beda karakter dari `pops.token`: staff instalasi perlu lihat/copy nilainya berulang kali saat setting modem, jadi **disimpan plain string** (kolom migration `string(6)`, bukan `text` + cast `encrypted` seperti `pops.token`) — enkripsi di sini cuma akan merepotkan tanpa manfaat nyata. **Auto-generate 6 digit acak** (`ServiceService::generatePin()`, `random_int` + `str_pad`) saat `create()` (field `pin` sengaja **tidak ada** di form create — `ServiceRequest` menandainya `nullable` untuk method selain PUT/PATCH), tapi **bisa diedit manual lewat form edit** (`ServiceRequest` mewajibkan `required|digits:6` saat PUT/PATCH) untuk staff reset PIN kalau perlu — `ServiceService::update()` mempertahankan `pin` lama kalau field edit dikosongkan (`$data['pin'] ?? $service->pin`, pola sama seperti `PopService::update()` mempertahankan `token`). Tidak ada validasi unique (bukan identitas global, cuma password lokal per service). Redirect setelah `store()` sengaja menyertakan PIN yang baru dibuat di flash `session('status')` supaya staff langsung tahu nilainya tanpa harus buka halaman edit lagi.
- **`residential_name`** — **bukan boolean**, tapi **nama perumahan/apartment/komplek** lokasi pemasangan (draft ERD menamainya `residential`, kolom yang salah diduga sebagai flag boolean di sesi perencanaan awal). **Di-rename ke `residential_name`** (string, nullable) supaya tidak membingungkan dibaca sebagai flag — pola yang sama seperti `packages.starter` → `is_starter`.
- **`deleted_at` pakai Eloquent `SoftDeletes` trait sungguhan** — **beda** dari semua modul lain (User/Product/Package/PoP/Coverage yang semua hard delete). Alasan: riwayat langganan pelanggan jarang boleh hilang permanen, beda dengan master data katalog/infrastruktur yang aman dihapus keras. `ServiceController::destroy()` (lewat `ServiceService::delete()`) jadi soft-delete — row tetap ada di DB, otomatis tersembunyi dari query default (index listing) berkat global scope `SoftDeletes`. Kalau menambah modul baru lain yang juga representasi riwayat/transaksi pelanggan (bukan master data), pertimbangkan pola yang sama; kalau cuma master data/katalog, tetap default ke hard delete seperti modul-modul sebelumnya.
- **`activated_at`/`expired_at`/`dismantled_at`/`canceled_at`** — kolom nullable timestamp biasa, **belum ada tombol/aksi transisi state machine** di iterasi ini (belum ada UI activate/dismantle/cancel) — sama seperti `pops.last_online_at` sebelum modul monitoring ada. Skema disiapkan duluan, logic-nya menyusul begitu modul Installation/Billing dibangun. Jangan improvisasi state machine/validasi transisi sebelum diminta eksplisit.
- **`status`** (string, default `pending_payment`) — kolom enum eksplisit (bukan dihitung dari kombinasi timestamp), konstanta di `App\Models\Service::STATUSES` (`pending_payment`, `pending_installation`, `active`, `suspended`, `canceled`, `dismantled`). Ditambahkan sebagai fondasi untuk alur bisnis siklus hidup service (registrasi → aktivasi → billing → suspend → dismantle) yang didiskusikan bareng user — lihat "Alur Bisnis Siklus Hidup Service" di bawah untuk konteks lengkap. **Belum ada UI/logic transisi status** (activate/suspend/dismantle/cancel) di iterasi ini, sama seperti kolom timestamp lifecycle di atas — `status` disiapkan duluan, cuma di-set `pending_payment` saat `ServiceService::create()`. Migration `2026_07_11_150001_add_status_to_services_table.php` juga backfill data lama dari kombinasi timestamp yang sudah ada (best-effort, karena riwayat timestamp tidak bisa membedakan `pending_payment` vs `pending_installation`).
- **Service baru otomatis membuat `Sale`** — `ServiceService::create()` (via `SaleService` yang di-inject lewat constructor) langsung membuat record `Sale` (order) untuk paket starter yang dipilih beserta `sale_products`-nya, disalin dari snapshot `package_product` paket tsb (`ServiceService::buildProductLines()`). Staff **tidak** input manual ke `/sales` terpisah untuk pendaftaran awal — ini menyambungkan langsung ke pembuatan tagihan pendaftaran di alur bisnis (lihat bawah). `ServiceService::update()` **tidak** ikut membuat/mengubah Sale (mengubah `package_id` lewat form edit Service tidak memicu order baru — itu skenario upgrade/renewal yang didokumentasikan sebagai belum diimplementasikan di section "Sales").
- **`user_id`** dibatasi harus user dengan **role `customer`** (bukan staff/superadmin) — divalidasi di `ServiceRequest::withValidator()` (query `User::find($id)->hasRole('customer')`, bukan rule bawaan Laravel karena butuh cek relasi role, bukan sekadar exists). Picker di form pakai **AJAX typeahead** (sama alasan seperti `subdistrict_id` di PoP — jumlah customer bisa besar) via endpoint `GET /services/customers/search` (`ServiceController::searchCustomers()`, scoped `User::role('customer')` dari `spatie/laravel-permission`, limit 20 JSON, digate `$this->authorize('viewAny', Service::class)`). **Query kosong (`q=''`) sengaja mengembalikan daftar browse (20 customer pertama diurutkan nama), bukan array kosong** — beda dari pola picker `subdistrict_id` yang cuma respons ke query non-kosong — supaya user bisa klik kolom kosong untuk langsung melihat daftar pelanggan (bukan wajib mengetik dulu), meniru pola picker service di `~/Webapp/xnet/app` yang jadi rujukan. Frontend (`services/_form.blade.php`) baru memulai pencarian bertahap (debounce 300ms) begitu user mengetik **minimal 3 karakter**; `@focus` pada kolom yang masih kosong (`customerId` belum terisi) langsung fetch daftar browse tanpa menunggu ketikan.

### Sales

Modul keenam yang sudah nyata — transaksi/order (`/sales`): mencatat produk apa saja yang dipesan untuk sebuah `service`, CRUD admin lewat Blade. Prasyarat (`services`) sudah lengkap sebelum modul ini dibangun; ini jadi bottleneck yang membuka Installation & Billing berikutnya (roadmap README menaruh `Installation → Billing → Invoice → Payment → Xendit` persis setelah `Service`). Scope iterasi ini **murni CRUD data transaksi** — TANPA integrasi Xendit/invoice/pembayaran (integrasi sungguhan dibangun belakangan sebagai modul terpisah, lihat "Billing / Invoice (Xendit)" di bawah).

- **`starter`** (nama kolom di draft ERD) → **di-rename `is_starter`** di migration, konsisten `packages.is_starter`. Kolom baru (tidak ada data lama yang bergantung ke nama lama), jadi tidak ada alasan menyimpang penamaan dari konvensi yang sudah ada. Nilainya **snapshot** dari `packages.is_starter` milik `package_id` yang sedang dipilih — **dihitung ulang di setiap `create()` maupun `update()`** (bukan dibekukan sejak insert pertama), supaya tidak stale kalau staff ganti `package_id` saat edit sale.
- **`total`** = SUM(`sale_products.price × quantity`) semua baris, **sebelum** diskon apa pun. **`subtotal`** = `total − discount`. **`grandtotal`** = `subtotal + tax + admin_fee`. Rumus ini dikonfirmasi eksplisit ke user sebelum ditulis (draft ERD punya dua kolom harga mirip — `total` dan `subtotal` — yang maknanya tidak dijelaskan di diagram).
- **`sales.discount`** = SUM(`sale_products.discount`) — **turunan otomatis server-side** dari seluruh baris produk (`SaleService::syncProductsAndRecalculate()`), **bukan** input manual staff terpisah, supaya tidak ada dua sumber kebenaran yang bisa tidak sinkron. `tax` dan `admin_fee` tetap input manual staff (nullable, default 0) — belum ada rate/config pajak yang bisa dihitung otomatis saat ini.
- **`sales.package_id`** independen dari `services.package_id`. `services.package_id` tetap mencatat paket pendaftaran awal (tidak berubah, sudah berjalan sejak modul Service). `sales.package_id` mencatat paket APA yang dipesan di transaksi itu spesifik — **tidak harus selalu sama** dengan `services.package_id` saat ini, mendukung skenario upgrade/renewal di masa depan (belum ada UI/logic upgrade eksplisit di iterasi ini, sekadar field yang sudah siap). Konsekuensinya: `<select>` `package_id` di form Sales menampilkan **SEMUA** paket (starter maupun bukan) — beda dari form Service yang membatasi cuma paket starter (lihat "Service" di atas).
- **Line item produk dipilih individual dari katalog `Product`** (bukan auto-generate dari isi `package_product`) — staff pilih produk satu-satu lewat picker dinamis Alpine.js di `sales/_form.blade.php`, replikasi pola `packages/_form.blade.php` (`rows`/`addRow()`/`removeRow()`/`applyDefaultPrice()`), tapi direstruktur jadi `<table>` dengan 5 kolom (produk/qty/harga/diskon/satuan — dua kolom terakhir baru, tidak ada di form Package) karena terlalu banyak untuk ditampilkan sebagai flex row tanpa label. Guard produk duplikat dalam satu submission pakai pola sama persis `PackageRequest::guardDuplicateProducts()`.
- **`total`/`subtotal`/`discount`/`grandtotal` tidak pernah diterima dari request** — field ini sengaja **tidak ada** di `SaleRequest::rules()` sama sekali, jadi `$request->validated()` tidak pernah menyertakannya walau form/klien mengirimnya. Nilai final selalu dihitung ulang penuh di `SaleService::syncProductsAndRecalculate()` dari `products[]` yang disubmit — prinsip "jangan percaya angka dari client". Preview total di form (`sales/_form.blade.php`) dihitung live via Alpine `x-text`/getter murni untuk kenyamanan input, tapi elemen preview itu **tidak punya `<input name="...">`** sama sekali sehingga tidak pernah ikut submit.
- **`service_id`** dipilih lewat **AJAX typeahead**, replikasi persis pola picker customer yang baru dibangun di form Service (`serviceQuery`/`serviceId`/`fetchServices`/`searchServices`/`openServiceBrowse`/`selectService`, debounce 300ms, ambang mengetik minimal 3 karakter, `@focus` pada kolom kosong langsung browse tanpa menunggu ketikan). Endpoint `GET /sales/services/search` (`SaleController::searchServices()`, digate `$this->authorize('viewAny', Sale::class)`) cari di `code`/`address`/nama atau nomor telepon pelanggan terkait (`whereHas('user', ...)`), limit 20, query kosong tetap kembalikan daftar browse (bukan array kosong). **Tidak perlu filter manual soft-delete di query builder ini** — `Service::query()` otomatis exclude row yang soft-deleted lewat global scope `SoftDeletes`, selama tidak memanggil `withTrashed()`. **Beda dengan `SaleRequest::rules()`**: `Rule::exists('services', 'id')->whereNull('deleted_at')` di sana **wajib eksplisit**, karena `exists` rule Laravel query langsung ke tabel dan tidak menghormati global scope Eloquent — tanpa filter eksplisit itu, staff bisa submit `service_id` yang sudah soft-deleted lewat request langsung (bypass picker) dan tetap lolos validasi.
- **`service_id`/`package_id` pakai `restrictOnDelete()`** (konsisten pola `coverage_id`/`pop_id` di `services`). Catatan: `services` sudah pakai `SoftDeletes`, jadi constraint di `sales.service_id` baru "bergigi" kalau ada `forceDelete()` — tetap dipasang sebagai lapisan pertahanan konsisten konvensi project, bukan sia-sia (test-nya sengaja pakai `$service->forceDelete()` langsung di level model untuk memicu ini, karena `DELETE /services/{id}` normal cuma soft-delete dan tidak pernah menyentuh FK).
- **`sale_products`** (pivot `sales`↔`products`, dikelola via `Sale::products(): BelongsToMany` + `withPivot(['price','discount','quantity','unit'])`, sama pola `Package::products()`) punya kolom tambahan `price`/`unit` (snapshot dari `products.price`/`products.unit` saat baris ditambahkan, independen dari katalog yang bisa berubah setelahnya — pola sama `package_product.price`) dan `discount` (diskon per-line, sumber `sales.discount` di atas). Unique constraint `['sale_id', 'product_id']` — satu produk cuma boleh muncul sekali per sale (pola sama `package_product`), naikkan `quantity` kalau butuh lebih dari 1 unit.
- **`invoiced_at`/`expired_at`/`settled_at`/`canceled_at`** — kolom nullable timestamp biasa, **belum ada tombol/aksi transisi state machine** di iterasi ini (belum ada UI invoice/settle/cancel) — pola sama persis `activated_at`/`dismantled_at` di Service. Skema disiapkan duluan, logic-nya menyusul begitu modul Billing/Xendit dibangun.
- **`deleted_at` pakai `SoftDeletes` sungguhan** — Sales = riwayat transaksi, alasan sama seperti Service (beda dari master data katalog/infrastruktur yang hard delete).
- **Tidak menambah relasi balik** (`Service::sales()`/`Package::sales()`) di iterasi ini — tidak ada UI/kebutuhan yang memakainya sekarang (mis. belum ada halaman "riwayat transaksi" di detail Service). Tambahkan nanti kalau benar-benar dibutuhkan, jangan didesain lebih dulu untuk kebutuhan hipotetis.
- **Authorization**: `App\Policies\SalePolicy` — sama seperti policy modul lain, seluruh 5 method cuma `return $user->isSuperadmin();`.
- **Service layer**: `app/Services/SaleService.php` — `create()`/`update()` isi `is_starter`/`code`(setelah insert)/audit trail, delegasikan hitung total & sync pivot ke `syncProductsAndRecalculate()` (method **publik**, bukan `private`, supaya bisa dipanggil ulang dari `SaleSeeder` tanpa `Auth::id()` yang `null` di konteks console — method ini sama sekali tidak menyentuh kolom audit); `delete()` (soft).

### Halaman Detail (Show) Lintas Modul

Ketujuh resource CRUD yang sudah ada (User, Product, Package, PoP, Coverage, Service, Sale) awalnya semua didaftarkan dengan `Route::resource(...)->except('show')` — cuma index/create/edit yang punya halaman, tidak ada halaman detail. Ini kemudian diminta eksplisit oleh user dan ditambahkan seragam ke semua 7 modul sekaligus (bukan bertahap), karena polanya identik persis lintas modul:

- **`->except('show')` dihapus** dari semua 7 baris `Route::resource()` di `routes/web.php`. Urutan route pencarian AJAX (`services/customers/search`, `sales/services/search`, `subdistricts/search`) tetap didaftarkan **sebelum** `Route::resource()` masing-masing supaya tidak tertelan wildcard `{service}`/`{sale}` — pola ini sudah benar sejak awal, tidak perlu diubah.
- **Controller `show()`** ditambahkan ke ketujuh controller, ditaruh di urutan konvensional Laravel (index, create, store, **show**, edit, update, destroy — bukan asal taruh di akhir file). Eager-load relasi mengikuti pola yang sudah dipakai `edit()` masing-masing controller (kadang diperluas, mis. `PopController::show()` juga load relasi balik `coverages` yang tidak di-load di `edit()`, karena halaman detail PoP menampilkan daftar coverage-nya).
- **Tidak ada perubahan Policy** — method `view()` di semua 7 Policy sudah ada sejak awal (`return $user->isSuperadmin();`, sama seperti `viewAny`/`create`/`update`/`delete`) karena `authorizeResource()` di constructor controller otomatis memetakan action `show` ke policy method `view`. Menambah `show()` ke controller langsung "bergigi" tanpa sentuhan lain.
- **Komponen Blade baru `<x-detail-row>`** (`resources/views/components/detail-row.blade.php`) — satu-satunya abstraksi baru yang ditambahkan, dipakai di ketujuh halaman show untuk baris label+value konsisten (grid `dt`/`dd`, dark mode pairing). Dibuat karena polanya identik dan terpakai lintas 7 file baru sekaligus (bukan spekulatif) — kalau nanti perlu varian lain (mis. baris dengan aksi inline), extend komponen ini, jangan bikin komponen show-detail kedua yang duplikatif.
- **View index** di ketujuh modul dapat tambahan link **"Detail"** (di sebelah "Ubah", warna netral abu-abu bukan `text-primary` supaya "Ubah" tetap jadi aksi utama yang menonjol) mengarah ke halaman show.
- **Halaman show saling bertaut** kalau relevan (mis. `services/show.blade.php` link ke `users.show`/`coverages.show`/`packages.show`; `sales/show.blade.php` link ke `services.show`/`packages.show`; `pops/show.blade.php` list coverage-nya link ke `coverages.show`) — memberi navigasi silang antar modul yang sebelumnya tidak ada sama sekali.
- **`pops.token` tetap tidak pernah ditampilkan** di halaman show PoP (konsisten `$hidden` + alasan keamanan yang sudah didokumentasikan di "PoP & Coverage" di atas).
- Test: satu test baru `test_superadmin_can_view_{modul}_detail...` ditambahkan ke masing-masing `tests/Feature/{Modul}ManagementTest.php` (bukan 403 test terpisah untuk `show` — gate `view()` policy sudah dianggap terverifikasi cukup lewat test 403 index yang sudah ada per modul, konsisten kedalaman test yang sudah berjalan sebelum perubahan ini).

### Alur Bisnis Siklus Hidup Service (Registrasi → Aktivasi → Billing → Suspend → Dismantle)

Sesi diskusi arsitektur (bukan modul kode baru) sebelum modul-modul besar berikutnya (Notification, Billing/Xendit, Installation, scheduler Suspend/Dismantle) mulai dibangun — user menjelaskan alur bisnis end-to-end, direview bareng, dan sebagian keputusan skema fondasinya (yang sudah cukup jelas untuk langsung dikerjakan) diimplementasikan di sesi yang sama. Modul-modul yang masih tersisa **belum** diimplementasikan, masing-masing butuh sesi desain sendiri sebelum ditulis jadi migration — mengikuti pola yang sudah berjalan untuk Service/Sales sebelumnya.

**Alur yang didiskusikan** (staff-facing, bukan customer-facing — customer akan pakai aplikasi terpisah yang terkoneksi lewat API, belum dibangun): staff tambah service baru → pilih/buat pelanggan → tagihan pendaftaran otomatis muncul (berlaku 3 hari, auto-batal kalau tidak dibayar) → pelanggan bayar via Xendit → teknisi diberi tahu untuk klaim & pasang → submit form instalasi → aktivasi otomatis, masa aktif dihitung dari paket → reminder H-5/H-3/H-1 sebelum expired → auto-suspend kalau telat bayar → auto-antre dismantle kalau suspend 2 bulan.

**Keputusan fondasi yang sudah diimplementasikan** (lihat detail per-kolom di section "Product & Package" dan "Service" di atas):
1. `products.type` tambah nilai `langganan` (1 qty = 1 bulan masa aktif kalau dibundel ke paket).
2. `packages.duration_months` — cache durasi, diturunkan dari quantity item `langganan` di paket (bukan dijumlah, harus seragam kalau lebih dari satu).
3. `services.status` — kolom enum eksplisit (`pending_payment`/`pending_installation`/`active`/`suspended`/`canceled`/`dismantled`), bukan cuma dihitung dari kombinasi timestamp.
4. `services.expired_at` **tetap kolom tersimpan** (bukan dihitung reaktif dari riwayat invoice seperti pola app sebelumnya `~/Webapp/xnet/app`) — nantinya diisi/diperpanjang langsung oleh service layer saat aktivasi & tiap renewal sukses. Konsisten dengan pola snapshot yang sudah dipakai NEXA (mis. `sales.discount`). **Logic pengisian/perpanjangannya sendiri belum ada** — itu bagian dari modul Installation/Billing yang menyusul.
5. Service baru otomatis membuat `Sale` (order) untuk paket starter yang dipilih — lihat "Service" di atas.

**Hal yang perlu diperhatikan sebelum modul-modul berikutnya digarap (belum diputuskan/berpotensi jadi blocker):**
- **Channel email untuk notifikasi teknisi butuh field email asli** — **sudah teratasi**: `email` sekarang field wajib asli di form `/users` (lihat "User" di atas), bukan placeholder lagi. Akun lama yang dibuat sebelum perubahan ini tetap punya placeholder sampai diedit ulang manual.
- **UX "tambah pelanggan baru inline dari form Service"** belum didesain — form Service saat ini cuma typeahead cari customer yang sudah ada, tidak ada opsi "buat baru". Rekomendasi (belum diimplementasikan): modal Alpine di atas form Service, bukan redirect ke `/users/create` lalu kembali, supaya draft form Service yang sudah terisi sebagian tidak hilang.
- Apakah `suspended_at` butuh kolom timestamp sendiri (atau cukup dihitung dari `expired_at` + `status = suspended`) — masih perlu dikonfirmasi ulang ke user saat modul Renewal digarap. (Efek auto-cancel & penanganan paket gratis/promo di poin sebelumnya **sudah** diputuskan & diimplementasikan — lihat "Billing / Invoice (Xendit)" di bawah.)

**Urutan pembangunan modul berikutnya yang disarankan** (masing-masing sesi implementasi terpisah): (1) fondasi skema — **sudah selesai**, lihat di atas; (2) **Notification module** — **sudah selesai**, lihat "Notification" di bawah; (3) **Billing/Invoice + integrasi Xendit** — **sudah selesai**, lihat "Billing / Invoice (Xendit)" di bawah; (4) **Installation workflow** — job queue teknisi, klaim pekerjaan, form instalasi (`service_activations` butuh kolom sungguhan, masih kerangka kosong di draft ERD), trigger aktivasi otomatis; (5) **Renewal + reminder H-5/H-3/H-1 + auto-suspend** — scheduled job (infrastruktur scheduler-nya sudah ada sejak modul Billing, tinggal menambah command baru); (6) **Dismantle queue** — scheduled job + workflow teknisi pembongkaran (`service_dismantles` juga masih kerangka kosong).

### Notification

Modul ketujuh yang sudah nyata — **murni infrastruktur** saat modul ini dibangun, belum ada satu pun trigger bisnis nyata yang memicunya. Modul ini cuma menyiapkan pipa (channel + UI + service wrapper) supaya begitu modul-modul berikutnya siap, mereka tinggal panggil `NotificationService::send()`. **Sekarang sudah ada 4 trigger bisnis nyata**, dibangun belakangan begitu modul yang memakainya jadi: `UserRegisteredNotification` (`UserService::create()`, lihat "User"), `ServiceRegisteredNotification` & `InvoiceCreatedNotification` (`ServiceService::create()`/`ReceiptService::createForSale()`, lihat "Service"), dan `PaymentReceivedNotification` (webhook Xendit, lihat "Billing / Invoice (Xendit)"). Reminder H-5/H-3/H-1 dan auto-suspend yang dibahas di "Alur Bisnis Siklus Hidup Service" di atas **masih tetap ditunda** ke modul "Renewal + reminder" terpisah — infrastruktur Laravel scheduler-nya sendiri **sudah ada** sejak modul Billing (`bootstrap/app.php` sudah pakai `withSchedule()`), jadi Renewal nanti tinggal menambah command baru, bukan membangun scheduler dari nol.

- **3 channel dibangun sekaligus**: in-app (tabel `notifications` bawaan Laravel, via `php artisan notifications:table` — `App\Models\User` sudah pakai trait `Illuminate\Notifications\Notifiable` sejak awal, cuma belum pernah dipakai), WhatsApp (reuse `WhatsappGateway` yang sama dengan OTP), dan email (wired secara struktural). **`users.email` sekarang field asli & wajib** (lihat "User" di atas — bukan placeholder inert lagi) dan kredensial SMTP sungguhan sudah ada di `.env` (`mail.xplus.id`), jadi channel mail secara teknis **bisa** benar-benar mengirim sekarang — tapi belum ada satu pun notification class di codebase yang menyertakan `'mail'` di `via()`-nya (`TestNotification` iya, tapi itu murni alat uji manual, bukan trigger bisnis nyata; `UserRegisteredNotification`/`PaymentReceivedNotification` sengaja cuma `['database', WhatsappChannel::class]`, scope permintaan masing-masing eksplisit WhatsApp, bukan email).
- **`WhatsappGateway::sendOtp()` tidak diubah** — tetap spesifik OTP dengan template hardcoded. Ditambah method generik baru **`sendMessage(string $phone, string $message): bool`** di interface + kedua driver (`LogWhatsappGateway`/`HttpWhatsappGateway`), dipakai semua kebutuhan notifikasi non-OTP. Konsekuensi: `CapturingWhatsappGateway` (test double dulu didefinisikan inline di `tests/Feature/Auth/LoginFlowTest.php`) **direfactor jadi `Tests\Support\CapturingWhatsappGateway`** (`tests/Support/CapturingWhatsappGateway.php`) supaya dipakai ulang oleh `LoginFlowTest` dan `NotificationTest` tanpa duplikasi — kalau menambah method baru lagi ke `WhatsappGateway` di masa depan, kelas ini harus diupdate juga (constraint interface `implements`).
- **Custom channel `App\Notifications\Channels\WhatsappChannel`** — **tidak didaftarkan lewat provider**, cukup dirujuk sebagai **class-string langsung** di `via()` notification class (`WhatsappChannel::class`, bukan string pendek `'whatsapp'`) — ini pola resmi Laravel untuk custom channel, container otomatis resolve constructor `WhatsappGateway`-nya lewat binding yang sudah ada di `WhatsappServiceProvider`. Kalau bingung kenapa tidak ada binding channel eksplisit di provider manapun, ini alasannya.
- **`config/notification.php`** — cuma `queue_connection` (default `sync`, env `NOTIFICATION_QUEUE_CONNECTION`), replikasi persis pola `config/otp.php` dan alasan yang sama: dev tidak menjalankan `queue:work`, jadi job yang didispatch ke `QUEUE_CONNECTION` global (`database`) akan nyangkut di tabel `jobs` tanpa pernah diproses.
- **`App\Services\NotificationService::send(object $notifiable, Notification $notification)`** — **satu-satunya cara** modul lain seharusnya mengirim notifikasi (bukan panggil `$user->notify()` langsung). Alasan konkret, bukan spekulatif: waktu diuji manual di dev, `.env` ternyata punya kredensial SMTP asli (`mail.xplus.id`), dan karena `users.email` masih placeholder, channel mail benar-benar mencoba kirim lalu **bounce** (`550` dari server mail) — exception ini tidak tertangkap Laravel secara default dan menjalar sampai meng-crash pemanggil, padahal channel database & WhatsApp yang lebih dulu di `via()` sudah berhasil. `NotificationService::send()` membungkus `$notifiable->notify()` dengan try/catch: kegagalan channel apa pun (mail bounce, gateway WhatsApp down, dll) di-log sebagai warning (`Log::warning`), tidak pernah menjalar ke pemanggil — channel yang sudah berhasil duluan tetap valid. Method ini juga yang menerapkan `onConnection(config('notification.queue_connection'))` kalau notification-nya queueable, supaya pemanggil tidak perlu mengulang itu sendiri.
- **`App\Notifications\TestNotification`** + **`php artisan notify:test --user=ID` (atau `--phone=`, opsional `--message=`)** — satu-satunya cara memicu notifikasi contoh di iterasi ini, murni untuk verifikasi manual end-to-end (in-app + WhatsApp log + mail), sengaja **tidak** disambungkan ke event bisnis apa pun. `via()` return `['database', WhatsappChannel::class, 'mail']`.
- **Bell dropdown di header** (`resources/views/components/header.blade.php`) — dibangun dari nol menggantikan placeholder dekoratif lama (icon + dot merah statis tanpa interaksi). Alpine dropdown mirror pola dropdown profil di file yang sama (`x-data="{ open: false }"`, `@click.outside`, `x-show`+`x-transition`). List di-**server-render** langsung dari `auth()->user()->unreadNotifications` (bukan AJAX fetch), badge dot cuma muncul kalau ada unread. Aksi tandai-dibaca (single & all) pakai **form POST biasa** (bukan AJAX/fetch) — project belum punya pola mutasi-via-fetch sama sekali (AJAX yang ada cuma untuk search/typeahead GET di PoP/Service/Sales), form-POST-refresh konsisten dengan tombol "Keluar" yang sudah ada di file yang sama.
- **`App\Http\Controllers\NotificationController`** — `markAsRead()`/`markAllAsRead()`, scope otomatis ke `Auth::user()->notifications()`/`unreadNotifications` (relasi bawaan trait `Notifiable`). **Sengaja tidak ada Policy baru** (beda dari modul lain yang gate `isSuperadmin()`) — ini bukan otorisasi role, cuma kepemilikan data personal; mencoba tandai-dibaca notifikasi milik user lain menghasilkan **404** (`findOrFail` di relasi yang sudah di-scope, bukan 403). Route: `POST /notifications/{notification}/read` (`notifications.read`), `POST /notifications/read-all` (`notifications.read-all`), grup `auth`.
- Test: `tests/Feature/NotificationTest.php` — command mengirim ke 3 channel (database tersimpan, WhatsApp ter-capture, mail lewat `Notification::fake()`), user/phone tidak ditemukan → exit gagal tanpa membuat baris, mark-as-read/mark-all-as-read ter-scope ke user sendiri (user lain 404, user lain tidak ikut ter-mark), bell menampilkan notifikasi unread, dan `NotificationService::send()` tidak crash walau salah satu channel melempar exception (diuji pakai notification+channel dummy yang sengaja throw, didefinisikan di file test yang sama).
- Gap yang didokumentasikan (bukan lupa): tidak ada reminder H-5/H-3/H-1, tidak ada auto-suspend (scheduler infrastrukturnya sudah ada sejak modul Billing, tinggal menambah command), email channel belum benar-benar mengirim (menunggu field `users.email` asli), belum ada halaman index notifikasi penuh (cuma dropdown 10 terbaru di header), belum ada preferensi notifikasi per-user (semua notifikasi ke semua channel yang di-`via()`, tidak bisa di-opt-out).

### Billing / Invoice (Xendit)

Modul kedelapan yang sudah nyata — menyambungkan tagihan pendaftaran (`Sale`, sudah dibuat otomatis sejak modul Service) ke pembayaran sungguhan lewat Xendit: pembuatan Payment Request otomatis, webhook penerima status bayar, dan auto-cancel invoice yang lewat jatuh tempo. Ini bottleneck besar yang sudah lama tercatat sebagai gap ("`receipts` belum punya kolom spesifik Xendit... padahal Xendit adalah fokus utama") — lihat "Draft Desain Database" di bawah dan "Alur Bisnis Siklus Hidup Service" di atas untuk konteks alur lengkapnya.

- **`sales.invoiced_at`/`expired_at`/`settled_at`/`canceled_at`** — kolom ini sudah ada sejak modul Sales (dan sudah dirender di `sales/show.blade.php`) tapi sebelum modul ini **belum pernah diisi oleh kode manapun**. Modul Billing yang pertama kali benar-benar mengisinya — **tidak ada migration baru ke tabel `sales` atau `services`** untuk modul ini, keduanya sudah siap sejak sebelumnya.
- **Tabel `receipts` baru** (`App\Models\Receipt`) — representasi satu Payment Request Xendit untuk sebuah Sale. Kolom: `code` (`REC`+6 digit, pola sama modul lain), `sale_id` (FK `restrictOnDelete`, **unique** — satu Sale maksimal satu Receipt di iterasi ini, tidak ada retry setelah expired), `xendit_payment_request_id` (nullable unique — null berarti belum berhasil dibuat di Xendit), `amount` (snapshot `grandtotal` saat dibuat), `status` (mirror status mentah dari Xendit — `PENDING`/`REQUIRES_ACTION`/`SUCCEEDED`/`FAILED`/`EXPIRED` dst, **sengaja tidak diterjemahkan ke enum sendiri** supaya tidak ada mapping yang bisa basi kalau Xendit menambah status baru), `checkout_url` (text, link hosted checkout), `raw_response` (json, full response terakhir untuk audit/debug), `created_by`/`updated_by`. **Tidak** `SoftDeletes` — receipt adalah log transaksi sistem, bukan resource yang di-CRUD manual. `Sale::receipt(): HasOne` adalah **relasi balik pertama** yang ditambahkan ke `Sale` — sebelumnya sengaja ditunda ("tambahkan nanti kalau benar-benar dibutuhkan"), sekarang kebutuhannya konkret (UI Sale show menampilkan status pembayaran).
- **Trigger otomatis, bukan tombol manual** — `ServiceService::create()` memanggil `ReceiptService::createForSale()` tepat setelah `Sale` pendaftaran dibuat, konsisten alur bisnis "tagihan pendaftaran otomatis muncul" yang sudah didiskusikan. **Panggilan ini sengaja dipindah ke luar `DB::transaction()`** yang membungkus insert Service+Sale (`ServiceService::create()` sekarang mengembalikan hasil transaksi lebih dulu, baru memanggil `ReceiptService` di luar closure-nya) — supaya panggilan HTTP eksternal ke Xendit yang lambat/gagal tidak menahan lock database.
- **Sale gratis (`grandtotal <= 0`, mis. paket promo)** — `ReceiptService::createForSale()` skip Xendit sepenuhnya: langsung isi `sale.settled_at = now()`, **tidak ada row `receipts` sama sekali**, tidak ada panggilan HTTP.
- **Kegagalan panggilan Xendit tidak membatalkan Service/Sale yang sudah dibuat** — `ReceiptService::createForSale()` membungkus panggilan gateway dengan try/catch: gagal → `Log::error`, `sale.invoiced_at` tetap null (sinyal "belum berhasil ditagihkan"), row `Receipt` tetap ada (status `PENDING`, `xendit_payment_request_id` null) supaya retry memakai row yang sama (bukan insert baru — akan bentrok unique `sale_id`). **Retry manual** (khusus kasus gagal-panggil-Xendit ini, **bukan** retry setelah invoice expired — itu tidak didukung, staff harus bikin Service baru) lewat tombol "Buat Tagihan" di `sales/show.blade.php` → `POST /sales/{sale}/receipt/retry` (`sales.receipt.retry`) → `SaleController::retryReceipt()`, digate `authorize('update', $sale)`, cuma jalan kalau `invoiced_at`/`settled_at`/`canceled_at` masih semua null.
- **Service wrapper `app/Services/Billing/`** — interface `XenditGateway` (`createPaymentRequest(referenceId, amount, description, enabledMethods): array`) + driver sungguhan `Drivers/HttpXenditGateway` (raw `Http::` facade, **bukan SDK**, sesuai README/CLAUDE.md, `withBasicAuth(secret_key, '')` ke `POST /v3/payment_requests`), dibind di `App\Providers\XenditServiceProvider` (terdaftar di `bootstrap/providers.php`). **Beda dari pola `WhatsappGateway`**: tidak ada driver `log` — Whatsapp butuh itu karena gateway WA sungguhan belum tersedia di dev, sedangkan kredensial sandbox Xendit sudah dimiliki user sejak awal modul ini; untuk test, binding di-override langsung ke `Tests\Support\FakeXenditGateway` (dibind default secara global di `tests/TestCase.php::setUp()` supaya **tidak ada test manapun yang pernah membuat panggilan HTTP sungguhan ke Xendit** — override per-test kalau perlu perilaku spesifik seperti simulasi gagal).
- **Metode pembayaran** yang diaktifkan di tiap Payment Request: Virtual Account, QRIS, E-wallet, Over-the-counter (Indomaret/Alfamart) — daftar lengkapnya di `config('billing.payment_methods')`, bukan dipilih staff per-invoice; pelanggan memilih sendiri di halaman checkout hosted Xendit. **Catatan**: bentuk pasti payload `payment_method` untuk mengaktifkan ke-4 channel ini sekaligus dalam satu request **belum diverifikasi end-to-end** ke sandbox sungguhan (kredensial dev baru diisi user setelah modul ini selesai) — payload di `HttpXenditGateway` best-effort dari dokumentasi publik Payment Requests API v3, revisit begitu diuji nyata.
- **Webhook** `POST /webhooks/xendit` (top-level di `routes/web.php`, **di luar** group `guest`/`auth` — dipanggil server-ke-server oleh Xendit, bukan browser). **CSRF di-exclude** lewat `$middleware->validateCsrfTokens(except: ['webhooks/xendit'])` di `bootstrap/app.php` — ini **pengecualian CSRF pertama** di project ini. `App\Http\Controllers\Webhooks\XenditWebhookController::handle()`: verifikasi header `x-callback-token` sama persis dengan `config('services.xendit.webhook_token')` (`hash_equals`, tolak 403 kalau tidak cocok — **bukan** otorisasi lewat Policy/Gate, ini kredensial server-ke-server), cari `Receipt` by `xendit_payment_request_id` (404 kalau tidak ketemu), **idempotency**: kalau `receipt.status` sudah `SUCCEEDED` dan event masuk juga sukses, langsung `200` tanpa mengulang efek samping. Efek samping sukses: `sale.settled_at = now()`, `service.status = STATUS_PENDING_INSTALLATION` (**bukan** `active` — aktivasi baru terjadi setelah modul Installation/form instalasi teknisi dibangun, konsisten alur bisnis "bayar → teknisi klaim & pasang → baru aktif"), kirim `PaymentReceivedNotification` ke `service.user` lewat `NotificationService::send()`. Status lain (gagal/expired dari sisi Xendit) **cuma update `receipt.status`**, sengaja **tidak** menyentuh `sales.canceled_at`/`services.status` — satu-satunya jalur pembatalan tetap scheduler auto-cancel (poin berikutnya), supaya tidak ada dua sumber kebenaran soal kapan sesuatu dianggap batal.
- **`App\Notifications\PaymentReceivedNotification`** — `via()` = `['database', WhatsappChannel::class]`, **tanpa** `mail` (konsisten gap `users.email` placeholder yang sudah tercatat di modul Notification).
- **Scheduler Laravel dibangun dari nol di modul ini** (project sebelumnya sama sekali belum punya) — `bootstrap/app.php` sekarang pakai `->withSchedule(fn (Schedule $schedule) => $schedule->command('billing:cancel-expired-invoices')->hourly())`. `App\Console\Commands\CancelExpiredInvoices` (signature `billing:cancel-expired-invoices`) query `Sale` yang `invoiced_at` terisi tapi `settled_at`/`canceled_at` masih null dan `expired_at` sudah lewat → dalam `DB::transaction()` per baris: `sale.canceled_at = now()`, `service.status = STATUS_CANCELED` (satu Sale registrasi = satu Service, jadi Sale batal berarti Service-nya ikut batal; retry berarti staff bikin Service baru, bukan reuse Sale lama), `receipt.status = 'EXPIRED'` (housekeeping). Durasi jatuh tempo di `config('billing.invoice_ttl_days')` (default 3, env `BILLING_INVOICE_TTL_DAYS`). **Dev lokal butuh scheduler jalan supaya command ini benar-benar tereksekusi** — `composer.json` script `dev` ditambah proses `php artisan schedule:work` (concurrently bareng `serve`/`queue:listen`/`pail`/`npm run dev`); tanpa ini scheduler tidak pernah jalan di dev (tidak ada cron).
- **Config**: `config('services.xendit')` untuk kredensial (`secret_key`, `webhook_token`, `base_url`, pola persis blok `whatsapp` di file yang sama) — kredensial sandbox diisi manual oleh user ke `.env`, tidak pernah di-commit. `config/billing.php` untuk aturan bisnis (`invoice_ttl_days`, `payment_methods`) — terpisah dari kredensial, pola sama `config/otp.php`/`config/notification.php`.
- **Tidak ada halaman CRUD terpisah untuk `receipts`** — murni ditampilkan menempel di `sales/show.blade.php` (status Xendit, link checkout, tombol retry kalau perlu), sistem-generated, tidak ada create/edit manual.
- Test: `tests/Feature/Billing/ReceiptCreationTest.php` (trigger otomatis saat Service dibuat, skip Xendit untuk Sale gratis, retry setelah gagal panggil Xendit, `InvoiceCreatedNotification` terkirim lewat WhatsApp berisi `checkout_url`), `tests/Feature/Billing/XenditWebhookTest.php` (efek samping sukses + notifikasi, token tidak valid ditolak, payment_request_id tidak dikenal 404, duplicate webhook idempotent), `tests/Feature/Billing/CancelExpiredInvoicesTest.php` (membatalkan Sale+Service yang lewat jatuh tempo, tidak menyentuh yang belum invoiced/belum expired/sudah settled). `tests/Support/FakeXenditGateway.php` — test double yang dibind default global di `tests/TestCase.php`.
- **Gap yang didokumentasikan** (dicatat eksplisit di sesi perencanaan, konsisten pola project "tiap gap besar = sesi desain sendiri"): notifikasi/klaim teknisi setelah pembayaran (bagian modul **Installation**), reminder H-5/H-3/H-1 & auto-suspend (modul **Renewal** terpisah — scheduler dasarnya sudah tersedia dari modul ini, tinggal menambah command), retry invoice setelah expired (tidak didukung, staff bikin Service baru), payload `payment_method` multi-channel belum diverifikasi ke sandbox sungguhan (lihat poin "Metode pembayaran" di atas).

### Payment Gateway (Xendit)

- Menggunakan **Xendit Payment Requests API v3**.
- Integrasi menggunakan **raw HTTP call** — SDK resmi Xendit sengaja tidak digunakan.
- Seluruh pemanggilan Xendit wajib melalui service wrapper internal (tidak dipanggil langsung dari controller), agar integrasi raw-HTTP ini tetap mudah dipelihara dan di-test, serta penanganan webhook (verifikasi signature, idempotency) punya satu tempat yang jelas.
- **Sudah diimplementasikan** — lihat "Billing / Invoice (Xendit)" di atas untuk detail nyata (service wrapper `app/Services/Billing/`, webhook, scheduler auto-cancel).

### Dark mode

- Dark mode ikut preferensi sistem (`prefers-color-scheme`) secara default, tapi user bisa override manual lewat tombol toggle — permintaan eksplisit user supaya tidak dipaksa satu mode saja.
- Tailwind v4 dikonfigurasi pakai varian `dark:` berbasis **class**, bukan cuma media query: `@custom-variant dark (&:where(.dark, .dark *));` di `resources/css/app.css`. Ini wajib ada supaya toggle manual (menambah/menghapus class `dark` di `<html>`) benar-benar mengubah tampilan.
- `resources/views/components/theme-init.blade.php` — script kecil yang **harus** dirender paling awal di `<head>` (sebelum elemen lain), tugasnya membaca `localStorage.theme` (kalau user pernah memilih manual) atau fallback ke `matchMedia('(prefers-color-scheme: dark)')`, lalu toggle class `dark` di `<html>` sebelum browser sempat menggambar (mencegah flash tema salah). Juga listen ke perubahan preferensi sistem selama user belum override manual.
- `resources/views/components/theme-toggle.blade.php` — tombol Alpine yang toggle class `dark` + simpan pilihan ke `localStorage.theme` ('light'/'dark'). Dipasang di `header.blade.php` (untuk halaman ber-sidebar) dan pojok kanan-atas `auth-layout.blade.php` (halaman login/OTP, karena belum ada header di sana).
- Setiap komponen/halaman baru yang punya warna latar/teks eksplisit **wajib** ditambahi pasangan `dark:` yang sepadan (pola: `bg-white` → `+ dark:bg-gray-800`, `text-gray-900` → `+ dark:text-white`, `border-gray-300` → `+ dark:border-gray-700`, dst.) — jangan biarkan halaman baru cuma punya tampilan light mode. Sidebar (`sidebar.blade.php`) terkecuali: dia memang didesain gelap permanen ("dark-sidebar" ala Metronic) terlepas dari mode terang/gelap keseluruhan app.

### Logo mengikuti tema (light/dark)

`public/images/logo/` berisi dua varian logo (aset sementara dari `~/Webapp/xnet/app11`), keduanya PNG transparan berisi mark "X" saja (bukan logo dengan warna latar solid, meski namanya menyiratkan begitu):

- `logo-white-bg.png` — X berwarna **hitam**, didesain untuk dipasang di atas latar **terang**.
- `logo-black-bg.png` — X berwarna **putih**, didesain untuk dipasang di atas latar **gelap**.

Aturan pemakaian (dicontek dari pola yang sama di `~/Webapp/xnet/app/`, yang pakai trik CSS `dark:invert` pada satu logo hitam — di NEXA dipakai pendekatan dua file dengan toggle visibility karena kita sudah punya kedua varian siap pakai, bukan filter invert):

- **Area yang latarnya ikut berubah terang/gelap** (mis. `auth-layout.blade.php`, tempat logo duduk di atas `bg-gray-100 dark:bg-gray-900`) — render **kedua** `<img>`, satu dengan class `dark:hidden` (pakai `logo-white-bg.png`, utk light mode) dan satu lagi `hidden dark:block` (pakai `logo-black-bg.png`, utk dark mode).
- **Area yang latarnya selalu gelap terlepas dari tema app** (`sidebar.blade.php` dan header/nav gelap di `welcome.blade.php`, keduanya `bg-gray-900` permanen) — **selalu** pakai `logo-black-bg.png` (X putih) saja, tanpa toggle, karena latar belakangnya tidak pernah berubah jadi terang.

Kalau menambah logo di tempat baru, cek dulu apakah latar elemen tersebut ikut berubah dengan tema app atau tetap satu warna — itu yang menentukan perlu toggle dua gambar atau cukup satu varian tetap.
